1. Architecture — Why Snowflake Is Different

Storage-Compute Separation · Virtual Warehouses · Micro-Partitions · Caching
Snowflake architecture = మన ఊరి గోడౌన్ (godown) system లా ఉంటది.
Storage = the godown where all goods (data) live. Compute = loading docks. You can have 3 separate docks (ETL team, BI team, Data Science) all pulling from the same godown simultaneously. Nobody waits for anyone. Pay rent for godown space. Pay workers only when they're on shift. Postgres is like a kirana shop — stock and counter in the same tiny room. You can't add more counters without moving the whole shop.

Snowflake is a cloud-native data warehouse built on a fundamental insight that traditional databases missed: storage and compute don't need to live together.

In Postgres, your data lives on disk attached to the same machine that runs queries. Need more compute? You either scale vertically (bigger machine, expensive, has a ceiling) or horizontally (read replicas, eventual consistency, operational headache). In Snowflake, data lives in cheap cloud object storage (S3/Azure Blob/GCS). Queries run on independent clusters called virtual warehouses that spin up, execute, and auto-suspend. You can have 10 warehouses querying the same table simultaneously — zero locking, zero contention.

[ KEY INSIGHT — WHY THIS MATTERS FOR COST ]
ETL takes 4 hours at night? Spin up an L warehouse. Shut it down when done. BI queries during the day? Run on a separate M warehouse. Data scientist runs an ad-hoc query? Their own XS warehouse. Each team gets isolated compute. You pay only for the seconds each warehouse is active. No idle capacity. No "the ETL is blocking my dashboard" complaints.
Micro-Partitions: How pruning works without indexes

Snowflake stores data in immutable 50-500 MB micro-partitions — columnar chunks automatically created on ingestion. Every partition's metadata records min/max values for every column. When you query WHERE order_date > '2026-01-20', Snowflake reads the metadata (not data), finds partitions whose max date is before your threshold, and skips them entirely. Zero bytes scanned for irrelevant partitions.

This is automatic — no CREATE INDEX, no REINDEX, no DBA maintenance. But it only works well if data is loaded in roughly sorted order. If your ETL loads random dates, every partition spans the full date range → pruning becomes useless → every query scans everything. The fix: add a clustering key to enforce physical sort order.

Three Cache Layers (and when each helps)

1. Metadata Cache: Statistics about micro-partitions. Lives in the Service Layer. Used for query planning and partition pruning. Always active, zero cost.

2. Result Cache: If you run exactly the same query (byte-for-byte identical SQL) within 24 hours, Snowflake returns the cached result instantly. Zero compute cost. This is why dashboard tools that parameterize queries need care — WHERE date = '2026-05-27' and WHERE date = '2026-05-28' are different queries, different cache entries.

3. Warehouse Cache (SSD): Each warehouse node has local SSD. Recently accessed micro-partitions stay in SSD. Subsequent queries on the same warehouse get faster reads. Cache is lost when warehouse suspends (auto-suspend default: 10 minutes idle).

⚠ COMMON PITFALL
Adding more nodes (L → XL) doesn't fix bad SQL. A cartesian join on XL is still slow — just 4× more expensive. Always check query profile before sizing up.
💡 INTERVIEW FRAME
"Snowflake separates storage and compute so you can run concurrent workloads on shared data without contention. Virtual warehouses are independent clusters that start, run, and suspend — you pay only for active seconds."

2. Getting Data In — Ingestion Patterns

COPY INTO · Snowpipe · Snowpipe Streaming · Streams + Tasks · Dynamic Tables
Data ni Snowflake loki teeskuraavadam = రైతు బజారుకి పంట తీసుకొచ్చినట్టు.
COPY INTO = truckload arriving at the wholesale market once a day (batch). Snowpipe = small tempo arriving every hour as vegetables are harvested (near real-time). Streams = the market clerk noting every change in a ledger (CDC log). Dynamic Tables = the shelf that auto-restocks when new produce arrives.

⚠ PITFALL — COPY INTO + PURGE = DATA LOSS
COPY INTO is idempotent by default. Snowflake tracks loaded files — running the same command twice won't duplicate data. But if you use PURGE = TRUE, files are deleted from S3 after the load commits. If the load fails mid-way? Already-purged files are gone. Recommendation: use PURGE only after verifying row counts. Better: separate archiving step. 
[ LISTING 001: COPY INTO — BATCH LOAD FROM S3 ]
-- File format (tells Snowflake how to parse)
CREATE FILE FORMAT my_csv TYPE='CSV'
  FIELD_OPTIONALLY_ENCLOSED_BY='"'
  SKIP_HEADER=1 NULL_IF=('NULL','null','');

-- External stage (pointer to cloud storage)
CREATE STAGE my_s3 URL='s3://my-bucket/landing/'
  STORAGE_INTEGRATION=my_s3 FILE_FORMAT=my_csv;

-- Load: auto-decompresses gzip, validates, idempotent
COPY INTO raw.orders FROM @my_s3
  PATTERN='.*orders_.*\\.csv\\.gz'
  ON_ERROR='SKIP_FILE_3%'
  PURGE=TRUE;  -- ⚠ deletes from S3 after load!
🔑 IDEMPOTENCY
COPY INTO tracks which files it loaded via metadata. Running it twice on same stage = zero duplicates. This is the safety net for scheduled ETL.
📋 FDE: DATA PIPELINE
Ingestion patterns = "workflow mapping + data pipeline design." In an interview, frame it as: "I mapped the data flow, selected the right ingestion pattern for each source, built the pipeline."

3. SQL — The Patterns You Write Every Week

Window Functions · MERGE · CTEs · Query Optimization

These four patterns cover ~80% of production SQL. Master them and you're ahead of most candidates claiming "5 years Snowflake experience." Each pattern below answers: what problem it solves, where it sits in the pipeline, and when NOT to use it.

[ PATTERN 001: DEDUPLICATION — GET LATEST ROW PER ENTITY ]
-- You will write this EVERY DAY. Memorize it.
-- Problem: raw_orders has duplicates (same order_id loaded multiple times)
-- Solution: keep only the latest version per order_id

SELECT * FROM raw_orders
QUALIFY ROW_NUMBER() OVER (
  PARTITION BY order_id
  ORDER BY loaded_ts DESC
) = 1;

-- QUALIFY = WHERE for window functions
-- Snowflake + BigQuery support it. For other DBs, wrap in CTE + WHERE rn=1.
-- When NOT to use: if you need to keep ALL versions (SCD Type 2), use LAG() instead.
[ PATTERN 002: CHANGE DETECTION — CDC WITH LAG() ]
-- Problem: which customers changed their address?
-- Solution: compare current row to previous row per customer

SELECT customer_id, address, effective_date,
  LAG(address) OVER (
    PARTITION BY customer_id ORDER BY effective_date
  ) AS prev_address,
  CASE WHEN address IS DISTINCT FROM
    LAG(address) OVER (PARTITION BY customer_id ORDER BY effective_date)
    THEN 'CHANGED' ELSE 'SAME' END AS status
FROM customer_history;
-- Only rows with status='CHANGED' → feed into SCD Type 2 MERGE
-- When NOT to use: if source already provides CDC flags (use those instead)
[ PATTERN 003: MERGE — UPSERT (MATCHED→UPDATE, NOT MATCHED→INSERT) ]
MERGE INTO dim_customer AS t
USING stg_customer_updates AS s
  ON t.customer_id = s.customer_id
WHEN MATCHED THEN UPDATE SET
  t.name=s.name, t.address=s.address, t.updated_ts=CURRENT_TIMESTAMP()
WHEN NOT MATCHED THEN INSERT
  (customer_id, name, address, created_ts, updated_ts)
  VALUES (s.customer_id, s.name, s.address, CURRENT_TIMESTAMP(), CURRENT_TIMESTAMP());

-- MERGE is atomic (all-or-nothing). Max 3 WHEN clauses:
--   MATCHED → UPDATE | MATCHED → DELETE | NOT MATCHED → INSERT
-- When NOT to use: if you only INSERT (use INSERT), only UPDATE (use UPDATE)
[ PATTERN 004: CTE PIPELINE — STAGE-BY-STAGE TRANSFORMATION ]
WITH
source AS (
  SELECT * FROM raw.orders WHERE status IS NOT NULL AND amount > 0
),
deduped AS (
  SELECT * FROM source
  QUALIFY ROW_NUMBER() OVER (PARTITION BY order_id ORDER BY loaded_ts DESC) = 1
),
enriched AS (
  SELECT d.date_key, c.customer_key, p.product_key, o.amount, o.qty
  FROM deduped o
  LEFT JOIN dim_date d ON o.order_date = d.date
  LEFT JOIN dim_customer c ON o.customer_id = c.customer_key
  LEFT JOIN dim_product p ON o.product_id = p.product_key
)
SELECT date_key, customer_key, COUNT(*) AS orders, SUM(amount) AS revenue
FROM enriched GROUP BY date_key, customer_key;

-- CTEs make pipelines readable. Each block = one stage.
-- When NOT to use: if intermediate results are used by 3+ downstream queries
--   → materialize as a temp table instead (CTE runs once per reference)

4. Change Data Capture — The Heart of DE

Watermark · Streams + Tasks · Dynamic Tables · Which Approach When
CDC = బ్యాంకు పాస్‌బుక్ లాగా. ప్రతి transaction record చేస్తాం.
CDC = like a bank passbook. Every transaction recorded: what changed, when, and how to apply it. Three approaches — pick based on source capabilities and complexity requirements.
Theorem: Choose the simplest CDC approach that meets your needs.
More complexity = more code to maintain, more things to break. Start simple. Add complexity only when requirements demand it. A Dynamic Table that works is better than a Streams+Tasks pipeline that's overengineered.
Approach 1: Watermark CDC — Simplest possible

When: Source table has updated_ts column. Deletes handled by soft-delete.

SET last_run = '2026-05-28 00:00:00';
INSERT INTO stg.orders
SELECT * FROM source.orders
WHERE updated_ts > $last_run;
-- Then dedup: QUALIFY ROW_NUMBER() OVER (PARTITION BY order_id ORDER BY updated_ts DESC)=1

✅ Pros: Simple. Works anywhere. ❌ Cons: Deleted rows invisible. Requires updated_ts.

Approach 2: Streams + Tasks — Full CDC control

When: Need INSERT/UPDATE/DELETE tracking. Complex MERGE logic. SCD Type 2.

CREATE STREAM orders_stream ON TABLE raw.orders;
CREATE TASK cdc_task WAREHOUSE=etl_wh SCHEDULE='1 MINUTE'
WHEN SYSTEM$STREAM_HAS_DATA('orders_stream')
AS
  MERGE INTO dim.orders t USING orders_stream s ON t.order_id=s.order_id
  WHEN MATCHED AND s.METADATA$ACTION='DELETE' THEN DELETE
  WHEN MATCHED AND s.METADATA$ISUPDATE=TRUE THEN UPDATE SET
    t.status=s.status, t.updated_ts=CURRENT_TIMESTAMP()
  WHEN NOT MATCHED THEN INSERT (order_id,...) VALUES (s.order_id,...);

✅ Pros: Full CDC. Delete tracking. ❌ Cons: More code. Stream offsets to manage.

Approach 3: Dynamic Tables — Declarative, zero CDC code

When: Simple transformation. No complex MERGE logic needed. Just want the result.

CREATE DYNAMIC TABLE dim.orders
  TARGET_LAG='5 minutes' WAREHOUSE=etl_wh
AS SELECT o.order_id, d.date_key, c.customer_key, o.amount
   FROM raw.orders o
   LEFT JOIN dim.date d ON o.order_date=d.date
   LEFT JOIN dim.customer c ON o.customer_id=c.customer_key;

✅ Pros: One SQL statement. Snowflake handles everything. ❌ Cons: Less control. No custom MERGE.

STAR Answer: "Tell me about a time you built a CDC pipeline"

Situation: ETL team's nightly load was taking 4+ hours on a Postgres replica. BI dashboards were stale by morning.

Task: Migrate to a near-real-time CDC pipeline on Snowflake without disrupting existing reports.

Action: Set up COPY INTO for initial load, then Snowpipe for incremental files from S3. Added Streams+Tasks for CDC within Snowflake — tracking INSERT/UPDATE/DELETE on order and customer tables. Implemented SCD Type 2 for customer address changes so historical reports remained accurate.

Result: Dashboard freshness dropped from 4hr to 2min. BI team got their own virtual warehouse, isolating query load. Historical data accuracy improved — previously overwritten addresses were lost; now both old and new addresses are queryable.

⚡ DECISION RULE
Simple transform → Dynamic Table. Complex SCD logic → Streams+Tasks. External source, no CDC → Watermark.
🎯 FDE: AUDIT TRAIL
Streams + Tasks = "checkpointing + retry with exponential backoff." Every change tracked. Failed MERGEs re-run on next Task tick.

5. Data Modeling — Star Schema, SCD, Grain

Fact · Dimension · Slowly Changing Dimensions · Surrogate Keys · Grain
Data modeling = ఇల్లు కట్టేముందు బ్లూప్రింట్ వెయ్యడం లాంటిది. రూమ్స్ (dimensions) ఎక్కడ, వస్తువులు (facts) ఎక్కడ — ముందే plan చేస్తే, queries fast గా run అవుతాయి.
Data modeling = drawing a blueprint before building a house. Where rooms (dimensions) go, where items (facts) go — plan it first, queries run fast later.
ConceptDefinitionExample
Fact TableBusiness events. Numeric, additive measures. Every row = something happened.orders, shipments, page_views, sensor_readings
DimensionDescribes who, what, where, when of facts. Text-heavy. Slowly changing.customers, products, stores, dates, campaigns
GrainThe level of detail one row represents. "One row per ___" — most critical design decision."One row per order line item per day"
Surrogate KeyArtificial integer key generated in warehouse. NOT the source natural key.customer_key (auto-inc) vs customer_id (from CRM)
SCD Type 1Overwrite old value. No history kept. Simple. Permanent data loss.Fix typo in customer name. Old value gone.
SCD Type 2Add new row. Track with valid_from/valid_to + is_current. Most common in practice.Customer changes address. Both old and new kept for historical reporting.
SCD Type 3Add column for previous value. Only one version of history. Rarely used.previous_address column. Only last change tracked.
⚠ MOST COMMON MODELING MISTAKE
Wrong grain. Designer thinks "one row per order" but the business needs "one row per order line item." Fixing this after 6 months of data → rewrite every downstream model. Always confirm grain with stakeholders before building anything.
💡 INTERVIEW: GRAIN
"The grain is the contract. Every fact table row represents exactly one {something}. If stakeholders disagree about what that something is, you have a design problem, not a SQL problem."

6. dbt — The T in ELT

Models · ref() · Materializations · Incremental Strategy
dbt = హోటల్ కిచెన్. Raw ingredients (raw data) → prep (staging) → cooking (intermediate) → plating (marts).
dbt = restaurant kitchen. Raw ingredients arrive → prep station (staging: clean, filter) → cooking (intermediate: joins, business logic) → plating (marts: analytics-ready tables). dbt manages the recipe book, not the stove — all transformations are SQL SELECTs.

dbt doesn't move data. It doesn't extract or load. It transforms data already inside Snowflake using pure SQL. Every model is a .sql file containing one SELECT statement. dbt compiles them, builds a dependency graph from {{ ref() }} calls, and executes in the correct order.

[ LISTING 005: DBT INCREMENTAL MODEL ]
{{
  config(materialized='incremental', unique_key='order_id')
}}
WITH source AS (
  SELECT * FROM {{ ref('stg_orders') }}
  {% if is_incremental() %}
    WHERE updated_ts > (SELECT MAX(updated_ts) FROM {{ this }})
  {% endif %}
),
enriched AS (
  SELECT o.order_id, d.date_key, c.customer_key,
         o.amount, o.qty, o.amount*o.qty AS line_total
  FROM source o
  LEFT JOIN {{ ref('dim_date') }} d ON o.order_date = d.date
  LEFT JOIN {{ ref('dim_customer') }} c ON o.customer_id = c.customer_key
)
SELECT * FROM enriched
Materializations: When to use which

view: Re-computed on every query. No storage cost. Slow for complex models. Use for lightweight staging.

table: Rebuilt from scratch on every dbt run. Use for small models (<100K rows) that change completely.

incremental: Only processes new/changed rows. The standard for fact tables. Needs unique_key. Use for large tables where full rebuild is too expensive.

ephemeral: Injected as CTE into downstream models. Not a real DB object. Use for intermediate logic used in one downstream model only.

🔑 REF() = DEPENDENCY
ref() does two things: resolves the table name AND declares "this model depends on X." dbt builds the DAG from these declarations.
