1 Window Functions — The DE Superpower

Window function = ప్రతి అడ్డు వరుసకి, దాని "పొరుగు" అడ్డు వరుసల సందర్భం చూపించడం. GROUP BY లాగా కాదు — అక్కడ వరుసలు కలిసిపోతాయి. ఇక్కడ అలా కాదు.
Window function = for each row, show context from its "neighbouring" rows WITHOUT collapsing them. Unlike GROUP BY which merges rows, window functions keep every row intact and just add a computed column.

The Big 5 Window Functions

FunctionWhat it doesDE Use Case
ROW_NUMBER() Assigns unique number 1,2,3... per partition De-duplication — get latest record per entity.
Most-used pattern in all of DE.
LAG() / LEAD() Access previous/next row's value Change detection — compare current value to previous.
CDC: "did the address change?"
SUM() OVER Running total Aggregation without collapse — running totals, rolling windows
RANK() / DENSE_RANK() Rank rows with possible ties Top-N per group — best-selling products per category
COUNT() OVER Count rows in a window Group size without GROUP BY — total orders per customer on each row

Pattern 1: De-duplication (Daily Use)

🔑 This is THE most common SQL pattern in Snowflake DE. You will write this every single week.
-- Scenario: raw_orders has duplicates (same order_id loaded multiple times)
-- Goal: get the latest version of each order

WITH dedup AS (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY order_id           -- group by business key
      ORDER BY loaded_ts DESC         -- keep the latest
    ) AS rn
  FROM raw_orders
)
SELECT * FROM dedup WHERE rn = 1;

-- Snowflake shortcut (QUALIFY clause — use this):
SELECT *
FROM raw_orders
QUALIFY ROW_NUMBER() OVER (
  PARTITION BY order_id
  ORDER BY loaded_ts DESC
) = 1;

-- QUALIFY = WHERE for window functions. Only Snowflake/BigQuery support it.
-- If your SQL tool doesn't support QUALIFY, use the CTE version.

Pattern 2: Change Detection (CDC)

-- Scenario: Detect which customers changed address
-- Compare current address to previous address

SELECT customer_id, address, effective_date,
  LAG(address) OVER (
    PARTITION BY customer_id
    ORDER BY effective_date
  ) AS prev_address,
  CASE
    WHEN address IS DISTINCT FROM LAG(address) OVER (
      PARTITION BY customer_id ORDER BY effective_date
    ) THEN 'CHANGED'
    ELSE 'SAME'
  END AS status
FROM customer_history;

-- Only rows where status = 'CHANGED' need SCD Type 2 processing
-- This directly feeds your MERGE statement

Pattern 3: Running Aggregations

-- Running total (revenue accumulating over time)
SELECT order_date, daily_revenue,
  SUM(daily_revenue) OVER (
    ORDER BY order_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS running_total
FROM daily_sales;

-- Moving average (7-day rolling)
SELECT order_date, daily_revenue,
  AVG(daily_revenue) OVER (
    ORDER BY order_date
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS rolling_7day_avg
FROM daily_sales;

-- Partitioned: running total per customer
SELECT customer_id, order_date, amount,
  SUM(amount) OVER (
    PARTITION BY customer_id
    ORDER BY order_date
  ) AS customer_running_total
FROM orders;

Pattern 4: Top-N Per Group

-- Top 3 products by revenue per category
WITH ranked AS (
  SELECT category, product_id, revenue,
    DENSE_RANK() OVER (
      PARTITION BY category
      ORDER BY revenue DESC
    ) AS rank
  FROM product_sales
)
SELECT * FROM ranked WHERE rank <= 3;

-- DENSE_RANK vs RANK:
-- DENSE_RANK: 1,2,2,3  (no gaps after ties)
-- RANK:       1,2,2,4  (skips 3 — gaps after ties)
-- Use DENSE_RANK for Top-N (you want 1,2,3 not 1,2,4)

2 MERGE — The UPSERT Pattern

MERGE = "ఉంటే అప్‌డేట్ చేయ్, లేకపోతే ఇన్సర్ట్ చేయ్". CDC పైప్ లైన్స్ లో ఇదే ప్రాణం.
MERGE = "If exists, UPDATE. If not, INSERT." This is the heartbeat of CDC pipelines.

Basic MERGE — Type 1 (Overwrite)

MERGE INTO dim_customer t           -- target table
USING stg_customer_updates s         -- source (new data)
  ON t.customer_id = s.customer_id   -- match condition

WHEN MATCHED THEN UPDATE SET         -- exists → update
  t.name = s.name,
  t.address = s.address,
  t.updated_ts = CURRENT_TIMESTAMP

WHEN NOT MATCHED THEN INSERT         -- new → insert
  (customer_id, name, address, created_ts, updated_ts)
  VALUES (s.customer_id, s.name, s.address, CURRENT_TIMESTAMP, CURRENT_TIMESTAMP);
✅ Key insight: MERGE is atomic — it's a single transaction. Either all changes apply, or none do. This is why it's used for CDC instead of separate INSERT/UPDATE statements.

MERGE — SCD Type 2 (History Tracking)

-- More complex: close old record, insert new one
MERGE INTO dim_customer t
USING stg_customer_updates s
  ON t.customer_id = s.customer_id
  AND t.is_current = TRUE             -- only match current records

WHEN MATCHED AND (
  t.address IS DISTINCT FROM s.address  -- did address change?
  OR t.phone IS DISTINCT FROM s.phone
) THEN UPDATE SET
  t.valid_to = CURRENT_DATE,          -- close old record
  t.is_current = FALSE                -- mark as history

OUTPUT s.customer_id, s.name, s.address, s.phone,
  CURRENT_DATE AS valid_from, NULL AS valid_to, TRUE AS is_current
  INTO dim_customer;                  -- insert new record

-- Snowflake doesn't support MERGE ... OUTPUT INTO directly.
-- Alternative: Use INSERT after MERGE with the changed keys.
-- Or use Snowflake Streams + Tasks for CDC.

For Snowflake specifically — use Streams + Tasks instead of complex MERGE:

-- Simpler approach with Streams
CREATE STREAM customer_changes ON TABLE stg_customer;

CREATE TASK scd_type2_task
  WAREHOUSE = my_wh
  SCHEDULE = '5 MINUTE'
  WHEN SYSTEM$STREAM_HAS_DATA('customer_changes')
  AS
    -- Close current records for changed customers
    UPDATE dim_customer
    SET valid_to = CURRENT_DATE, is_current = FALSE
    WHERE customer_id IN (
      SELECT customer_id FROM customer_changes
      WHERE METADATA$ACTION = 'INSERT'  -- new data = change
    )
    AND is_current = TRUE;

    -- Insert new records
    INSERT INTO dim_customer (customer_id, name, address, valid_from, is_current)
    SELECT customer_id, name, address, CURRENT_DATE, TRUE
    FROM customer_changes
    WHERE METADATA$ACTION = 'INSERT';

MERGE with Delete Handling

MERGE INTO fact_sales t
USING stg_sales_updates s
  ON t.sale_id = s.sale_id

WHEN MATCHED AND s.operation = 'DELETE' THEN DELETE

WHEN MATCHED THEN UPDATE SET
  t.amount = s.amount,
  t.qty = s.qty

WHEN NOT MATCHED THEN INSERT
  (sale_id, amount, qty)
  VALUES (s.sale_id, s.amount, s.qty);
✅ Remember: You can have up to 3 WHEN clauses in a MERGE: MATCHED UPDATE, MATCHED DELETE, NOT MATCHED. The order matters — evaluate top to bottom.

3 CTEs — Clean Pipeline Logic

CTE = చిన్న చిన్న బ్లాక్స్ గా SQL ని రాయడం. ప్రతి బ్లాక్ ఒక దశ (stage). చివర్లో SELECT వేసి అన్నీ కలిపేయడం.
CTE = writing SQL as small, named blocks. Each block is one stage of the pipeline. Final SELECT assembles everything.

CTE Pipeline — dbt-Style

This is how dbt models are structured internally. Think of it as: source → stage → intermediate → final

WITH
-- Stage 1: Raw data quality filter
source AS (
  SELECT * FROM raw_orders
  WHERE status IS NOT NULL
    AND amount > 0
),

-- Stage 2: Deduplicate (CDC pattern)
deduped AS (
  SELECT * FROM source
  QUALIFY ROW_NUMBER() OVER (
    PARTITION BY order_id ORDER BY loaded_ts DESC
  ) = 1
),

-- Stage 3: Enrich with dimensions
enriched AS (
  SELECT
    d.date_key,
    c.customer_key,
    p.product_key,
    o.amount,
    o.qty,
    o.status
  FROM deduped o
  LEFT JOIN dim_date d ON o.order_date = d.date
  LEFT JOIN dim_customer c ON o.customer_id = c.customer_key
  LEFT JOIN dim_product p ON o.product_id = p.product_key
),

-- Stage 4: Aggregate for reporting
aggregated AS (
  SELECT
    date_key,
    customer_key,
    COUNT(*) AS order_count,
    SUM(amount) AS total_revenue,
    SUM(qty) AS total_units
  FROM enriched
  GROUP BY date_key, customer_key
)

SELECT * FROM aggregated;
📌 CTE vs Temp Table vs Subquery:
• CTE — readable, refactorable, good for 2-4 stages. Snowflake materializes CTEs only when needed.
• Temp table — materialized on disk. Use when downstream needs the same intermediate data.
• Subquery — hard to read, hard to debug. Avoid for multi-stage logic.

Recursive CTE — Employee Hierarchy

-- Snowflake supports recursive CTEs (not commonly needed, but useful for org charts)
WITH RECURSIVE org_tree AS (
  -- Anchor: top-level managers
  SELECT employee_id, employee_name, manager_id, 1 AS level
  FROM employees
  WHERE manager_id IS NULL

  UNION ALL

  -- Recursive: direct reports
  SELECT e.employee_id, e.employee_name, e.manager_id, t.level + 1
  FROM employees e
  JOIN org_tree t ON e.manager_id = t.employee_id
)
SELECT * FROM org_tree ORDER BY level, employee_name;

4 CDC Pattern Language

CDC = డేటాలో మార్పులని ట్రాక్ చేసే పద్ధతి. "ఏమి మారింది, ఎప్పుడు మారింది, ఎలా మారింది" అనే మూడు ప్రశ్నలకి సమాధానం.
CDC = Change Data Capture. Tracks what changed, when, and how. The three questions every CDC system answers.

CDC Pattern 1: Watermark-Based

-- Source table has a last_updated timestamp
-- Extract only rows that changed since last run

SET last_run = '2026-05-26 12:00:00';

INSERT INTO stg_orders
SELECT *
FROM source_orders
WHERE updated_ts > $last_run;

-- Deduplicate (in case same row was updated multiple times)
SELECT *
FROM stg_orders
QUALIFY ROW_NUMBER() OVER (
  PARTITION BY order_id
  ORDER BY updated_ts DESC
) = 1;

Pros: Simple, works everywhere. Cons: Deleted rows are invisible (unless soft-delete). Requires updated_ts column on source.

CDC Pattern 2: Snowflake Streams

-- Stream = CDC log on a Snowflake table
-- Tracks INSERT, UPDATE, DELETE with BEFORE/AFTER values

CREATE OR REPLACE STREAM orders_stream ON TABLE raw_orders;

-- Read the changes (this also advances the stream offset)
SELECT *,
  METADATA$ACTION AS action,       -- 'INSERT', 'DELETE', 'UPDATE'
  METADATA$ISUPDATE AS is_update,  -- TRUE if this is an UPDATE
  METADATA$ROW_ID                  -- unique row identifier
FROM orders_stream;

-- Automate with a TASK
CREATE OR REPLACE TASK process_orders_cdc
  WAREHOUSE = my_wh
  SCHEDULE = '1 MINUTE'
  WHEN SYSTEM$STREAM_HAS_DATA('orders_stream')
  AS
    MERGE INTO dim_orders t
    USING orders_stream s
      ON t.order_id = s.order_id

    WHEN MATCHED AND s.METADATA$ACTION = 'DELETE' THEN DELETE

    WHEN MATCHED AND s.METADATA$ISUPDATE = TRUE THEN UPDATE SET
      t.status = s.status,
      t.amount = s.amount,
      t.updated_ts = CURRENT_TIMESTAMP

    WHEN NOT MATCHED THEN INSERT
      (order_id, customer_id, amount, status, created_ts, updated_ts)
      VALUES (s.order_id, s.customer_id, s.amount, s.status,
              CURRENT_TIMESTAMP, CURRENT_TIMESTAMP);
✅ Key: Streams show changes since the last read. Multiple consumers? Multiple streams. Each stream maintains its own offset.

CDC Pattern 3: Dynamic Tables (Simpler Alternative)

-- Dynamic Table = declare the SQL you want, Snowflake maintains it
-- No streams, no tasks, no MERGE — just SQL

CREATE OR REPLACE DYNAMIC TABLE dim_orders
  TARGET_LAG = '5 minutes'
  WAREHOUSE = my_wh
AS
  SELECT
    o.order_id,
    d.date_key,
    c.customer_key,
    o.amount,
    o.status,
    o.updated_ts
  FROM raw_orders o
  LEFT JOIN dim_date d ON o.order_date = d.date
  LEFT JOIN dim_customer c ON o.customer_id = c.customer_id;

-- Snowflake automatically:
-- 1. Detects changes in raw_orders
-- 2. Incrementally updates dim_orders
-- 3. Keeps it within 5 minutes of real-time
-- No MERGE, no Task, no Stream boilerplate.

When to use: Dynamic Tables = less control, less code. Streams + Tasks = full control, more code. Start with Dynamic Tables for simple transformations. Use Streams + Tasks when you need SCD logic or complex MERGE behavior.

5 Query Optimization — Write Fast SQL

Slow query = డబ్బు నష్టం. Snowflake pay-per-query. పనికి ముందు ఒకసారి ఆలోచించు — ఎంత డేటా స్కాన్ అవుతుంది? ఎలా తగ్గించాలి?
Slow query = money lost. Snowflake charges per query executed. Think before you write — how much data will this scan? How do I reduce it?

The "Avoid vs Prefer" Table

❌ Avoid 
SELECT * FROM big_table
WHERE YEAR(date) = 2026
AND MONTH(date) = 5

SELECT DISTINCT col
FROM table

SELECT *
FROM a, b
WHERE a.id = b.id

SELECT *
FROM big_table
ORDER BY date

-- Subquery in WHERE
SELECT * FROM t
WHERE id IN (
  SELECT id FROM huge_table
)
✅ Prefer 
SELECT col1, col2 FROM big_table
WHERE date >= '2026-05-01'
  AND date < '2026-06-01'
-- WHY: YEAR() wraps filter — no partition pruning

SELECT col, COUNT(*) FROM table
GROUP BY col
-- WHY: DISTINCT + scan no aggregation vs GROUP BY

SELECT * FROM a
JOIN b ON a.id = b.id
-- WHY: Explicit JOIN, no accidental cartesian

SELECT * FROM big_table
WHERE date >= '2026-01-01'
ORDER BY date
-- WHY: WHERE first reduces rows before sort

SELECT * FROM t
WHERE EXISTS (
  SELECT 1 FROM huge_table h
  WHERE h.id = t.id
)
-- WHY: EXISTS stops at first match, IN scans all

The ORDER of Operations (Mental Model)

SQL doesn't execute in the order you write it. Internally, the engine does:

1. FROM / JOIN      → which tables, how to join
2. WHERE            → filter rows (EARLY = good)
3. GROUP BY         → collapse groups
4. HAVING           → filter groups
5. WINDOW functions → compute windowed columns
6. QUALIFY          → filter window results
7. SELECT           → pick columns
8. DISTINCT         → deduplicate
9. ORDER BY         → sort (LAST — expensive)
10. LIMIT           → take N rows
📌 Strategy: WHERE filters rows BEFORE GROUP BY. So filter as early as possible. A WHERE date >= '2026-01-01' means GROUP BY works on fewer rows, which is faster and cheaper.

Snowflake-Specific Optimization Tips

TipWhy
Filter on clustered columnsSnowflake can skip entire micro-partitions when WHERE uses the clustering key. Without it, scans every partition.
Avoid SELECT * in prodSnowflake stores data columnar. Selecting 3 columns vs 30 = 10x less data scanned = 10x cheaper.
Use LIMIT during developmentBefore running the full query, test with LIMIT 1000. Costs pennies instead of dollars.
Materialize intermediate resultsIf a CTE is used by multiple downstream queries, make it a temp table. CTE runs once per reference.
Check BYTES_SCANNED in query historyThis is your cost metric. Each TB scanned ≈ $X. Reduce bytes scanned = reduce cost.
Use LIKE wiselyLIKE '%text' can't use partitioning (leading wildcard). LIKE 'text%' can.
-- Check your query's bytes scanned
SELECT query_id, query_text,
  bytes_scanned, bytes_scanned / 1e12 AS terabytes_scanned,
  total_elapsed_time / 1000 AS seconds
FROM TABLE(INFORMATION_SCHEMA.QUERY_HISTORY())
WHERE query_id = '<your-query-id>';

-- Target: bytes_scanned should be close to the size of data you actually need
-- If bytes_scanned >> expected, partition pruning isn't working

6 SQL Check

1. You have a table with duplicate order_ids (same order loaded multiple times with different timestamps). Which SQL pattern gets only the latest version of each order?
✅ Correct! ROW_NUMBER() with PARTITION BY order_id and ORDER BY loaded_ts DESC assigns 1 to the newest row. QUALIFY filters to keep only that row. DISTINCT and GROUP BY can't pick the "latest" version — they just collapse duplicates arbitrarily.
2. A customer changes their address. You need to keep historical addresses in dim_customer. What's the right SQL pattern?
✅ Correct! SCD Type 2 = close the current record (set valid_to, is_current=FALSE), insert a new record with valid_from=today. This preserves the history for reporting on "what was the address at the time of the order."
3. Which query will scan LESS data in Snowflake?
✅ Correct! Query B uses a range filter directly on the column, which allows Snowflake to do partition pruning. Query A wraps the column in YEAR() — Snowflake can't use the clustering key, so it scans EVERY partition, applies YEAR() to every row, THEN filters.
4. In Snowflake, what's the difference between a Stream and a Dynamic Table?
✅ Correct! Stream = raw CDC data (you write the MERGE logic). Dynamic Table = you write the final SQL, Snowflake handles the incremental maintenance. Streams give more control but need more code. Dynamic Tables are simpler but less flexible.
5. You notice a query is scanning 10x more data than expected. What's the first thing to check?
✅ Correct! First check query profile → bytes scanned vs partitions scanned. If partitions scanned = total partitions, your WHERE isn't pruning. Look for YEAR(date), MONTH(date), or other function-wrapped columns. Fix: use range filter directly on the column instead.
