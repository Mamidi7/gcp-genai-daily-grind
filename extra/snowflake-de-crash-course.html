1 Big Picture — Pipeline Architecture

This is the end-to-end flow. Every project you build will fit into this picture. Know which box you're working in.

2 Snowflake Core Concepts

Snowflake = ఒక పెద్ద గోడౌన్, కానీ కంప్యూట్ మరియు స్టోరేజ్ వేరు. నీకు కావల్సినప్పుడు మాత్రమే కంప్యూట్ కి డబ్బు.
Snowflake = a massive warehouse where storage and compute are separate. You pay for storage always, compute only when running queries.

Storage vs Compute Separation

LayerWhat it isPays
StorageCompressed, columnar, encrypted data on S3. Micro-partitions automatically optimized. Data is always there — even when no queries run.Per TB per month
ComputeVirtual warehouses = clusters of VMs that run SQL. Can start/stop/resize independently. Multiple warehouses can query the same data simultaneously.Per second while running
ServicesCloud services layer — authentication, metadata, query parsing, optimization. Always on, always free.Free (included)

Virtual Warehouses — The Engine

What they are: Clusters of compute resources (CPU, memory, local storage) that execute SQL queries.

Types: X-Small → 2X-Large → 6X-Large (each step = 2x the resources)

Key behaviors:

  • Multi-cluster: For concurrency. Auto-scales from 1 to N clusters when queue builds up.
  • Auto-suspend: After N minutes of idle, warehouse stops. No cost when not running.
  • Auto-resume: Query comes in → warehouse starts automatically (3-5 sec delay).
  • Cache: Each warehouse has its own result cache. Same query from same warehouse ≈ instant.
Scale Up = larger warehouse = faster single query Scale Out = more clusters = handle more concurrent users

Micro-Partitions & Clustering

Micro-partitions: Snowflake automatically splits your table data into tiny, compressed columnar chunks (50-500 MB each). No manual partitioning needed — it's automatic on insert.

Clustering keys: Optional. Tell Snowflake which column to sort micro-partitions by. Helps queries that filter by that column scan fewer partitions.

-- Create a clustering key on a large table
ALTER TABLE orders CLUSTER BY (order_date);

-- Check clustering depth
SELECT SYSTEM$CLUSTERING_DEPTH('orders');
-- Lower depth = better clustering = fewer partitions scanned

Use clustering keys only on tables > 1 TB with frequent filtering on a specific column. Not needed for small tables.

Caching Layers (Know These)

3 Data Modeling — Star Schema

స్టార్ స్కీమా = మధ్యలో ఒక ఫ్యాక్ట్ టేబుల్ (అమ్మకాలు), చుట్టూ డైమెన్షన్ టేబుల్స్ (తేదీ, కస్టమర్, ప్రొడక్ట్). నక్షత్రం లాగా.
Star schema = one fact table in the center (measurements), dimension tables around it (context). Looks like a star.

Fact Table — The "What Happened"

TypeUseGrainExample
Transaction FactEach event = one rowOne row per transactionsales — one row per item purchased
Periodic SnapshotRegular summaryOne row per perioddaily_account_balance — end-of-day snapshot
Accumulating SnapshotTrack a processOne row per entity over its lifecycleorder_fulfillment — ordered→shipped→delivered dates
Factless FactEvents with no measureOne row per occurrencestudent_attendance — present/absent, no numeric value

Golden rule: Always define the grain before designing a fact table. "One row = one booking transaction." If you don't define grain, the table has no discipline.

Dimension Table — The "Context"

Dimensions describe who, what, where, when. They are the labels and filters for your fact measurements.

Conformed dimensions — dimensions that are identical across multiple fact tables. Example: dim_date is the same for sales facts, inventory facts, and HR facts. This lets you combine data from different processes on the same date dimension.

-- Conformed dimension example
CREATE TABLE dim_date (
    date_key       INT PRIMARY KEY,  -- e.g., 20260527
    date           DATE,
    year           INT,
    quarter        INT,
    month          INT,
    month_name     VARCHAR(10),
    day_of_week    VARCHAR(10),
    is_weekend     BOOLEAN
);

-- Used identically by EVERY fact table in the warehouse
SELECT d.month, SUM(f.amount)
FROM fact_sales f
JOIN dim_date d ON f.date_key = d.date_key
GROUP BY d.month;

Slowly Changing Dimensions (SCD)

When a dimension attribute changes — a customer moves, a product gets renamed — how do you handle it?

TypeBehaviorAnalogyWhen
Type 0No change. Keep original."First version is forever."Date dimensions, immutable identifiers
Type 1Overwrite. Old value is gone."White-out on paper."Fix typos, current phone number (no need to track history)
Type 2Add new row with date range. Track history."Employee changed departments — keep both old and new with dates."Customer address, employee department, vendor status — most common
Type 3Add a previous-value column."Original value + current value side by side."Limited history (e.g., "previous territory" alongside "current territory")
-- SCD Type 2: dim_customer
-- When customer moves, INSERT a NEW row with valid_from/valid_to
-- Old rows get valid_to = current_date, is_current = FALSE

SELECT customer_key, address, valid_from, valid_to, is_current
FROM dim_customer
WHERE customer_business_key = 'CUST123'
ORDER BY valid_from DESC;
-- Returns: current row + all historical rows

4 Data Ingestion — Getting Data Into Snowflake

డేటా లోడ్ = సరుకు ట్రక్కు నుండి గోడౌన్ లోకి ఎక్కించడం. బ్యాచ్ (పెద్ద మొత్తం ఒకేసారి) లేదా స్ట్రీమింగ్ (కొద్ది కొద్దిగా వస్తూనే).
Data loading = moving goods from truck into warehouse. Batch (large loads at once) or streaming (continuous drip).
-- Step 1: Create stage (pointer to S3 location)
CREATE OR REPLACE STAGE my_s3_stage
  URL = 's3://my-bucket/landing/orders/'
  STORAGE_INTEGRATION = my_s3_int;

-- Step 2: Define file format
CREATE OR REPLACE FILE FORMAT csv_format
  TYPE = CSV
  SKIP_HEADER = 1
  FIELD_OPTIONALLY_ENCLOSED_BY = '"';

-- Step 3: COPY INTO
COPY INTO raw_orders
FROM @my_s3_stage
FILE_FORMAT = csv_format
PATTERN = '.*orders_.*\.csv'
ON_ERROR = 'CONTINUE';  -- skip bad rows, continue loading

Key points:

  • ON_ERROR = 'CONTINUE' — don't let one bad file kill the whole load
  • PURGE = TRUE — delete source file after successful load (optional)
  • VALIDATION_MODE = RETURN_ERRORS — preview errors before actually loading
  • Snowflake tracks which files have been loaded — COPY INTO is idempotent by default

Snowpipe = event-driven ingestion. When a file lands in S3, an SQS notification triggers Snowpipe, which loads it automatically.

-- Create pipe
CREATE OR REPLACE PIPE my_pipe
  AUTO_INGEST = TRUE
  AS
    COPY INTO raw_orders
    FROM @my_s3_stage
    FILE_FORMAT = csv_format;

-- Check pipe status
SELECT SYSTEM$PIPE_STATUS('my_pipe');
-- Returns: running, pending file count, last error

Cost tradeoff: Snowpipe is priced per MB loaded, not per warehouse credit. If you have continuous small files, Snowpipe is cheaper than running a warehouse 24/7.

Snowflake Streams = CDC (Change Data Capture) without external tools. A stream tracks inserts, updates, deletes on a table since the last read.

-- Create a stream on the source table
CREATE OR REPLACE STREAM orders_stream ON TABLE raw_orders;

-- Read changes (this advances the offset)
SELECT *
FROM orders_stream
WHERE METADATA$ACTION = 'INSERT';  -- or 'DELETE' or 'UPDATE'

-- Use with a TASK to automate
CREATE OR REPLACE TASK process_orders
  WAREHOUSE = my_wh
  SCHEDULE = '5 MINUTE'
  AS
    MERGE INTO dim_orders t
    USING orders_stream s ON t.order_id = s.order_id
    WHEN MATCHED AND s.METADATA$ACTION = 'DELETE' THEN DELETE
    WHEN MATCHED THEN UPDATE SET ...
    WHEN NOT MATCHED THEN INSERT (...);

Dynamic Tables (newer alternative): Declarative — define the SQL you want, Snowflake maintains the result incrementally. Less control, less boilerplate.

5 SQL You'll Actually Write

ఇక్కడ SQL నీ ప్రాణం. 80% పని = SQL. కింది ప్యాటర్న్స్ తప్పకుండా పట్టు.
SQL is your lifeblood here. 80% of the job is writing SQL. Master these patterns.
-- ROW_NUMBER: Get latest record per entity (CDC pattern)
SELECT *
FROM (
  SELECT *, ROW_NUMBER() OVER (
    PARTITION BY order_id ORDER BY updated_ts DESC
  ) AS rn
  FROM raw_orders
) WHERE rn = 1;  -- only the most recent version

-- LAG: Compare current row to previous
SELECT order_date, amount,
  LAG(amount) OVER (ORDER BY order_date) AS prev_day_amount,
  amount - LAG(amount) OVER (ORDER BY order_date) AS day_over_day_change
FROM daily_sales;

-- SUM OVER: Running total
SELECT order_date, amount,
  SUM(amount) OVER (ORDER BY order_date) AS running_total
FROM daily_sales;

-- RANK / DENSE_RANK: Top-N per group
SELECT product_id, revenue,
  DENSE_RANK() OVER (PARTITION BY category ORDER BY revenue DESC) AS rank_in_cat
FROM product_sales;
-- MERGE: UPSERT pattern (core of CDC processing)
MERGE INTO dim_customer t
USING stg_customer_updates s ON t.customer_id = s.customer_id
WHEN MATCHED AND t.address != s.address THEN
  -- Close current record, insert new (SCD Type 2)
  UPDATE SET t.valid_to = CURRENT_DATE, t.is_current = FALSE
WHEN NOT MATCHED THEN
  INSERT (customer_id, name, address, valid_from, valid_to, is_current)
  VALUES (s.customer_id, s.name, s.address, CURRENT_DATE, NULL, TRUE);

-- Alternative: Simple upsert with overwrite
MERGE INTO fact_sales t
USING stg_sales s ON t.sale_id = s.sale_id
WHEN MATCHED THEN UPDATE SET
  t.amount = s.amount, t.qty = s.qty, t.updated_ts = CURRENT_TIMESTAMP
WHEN NOT MATCHED THEN INSERT (sale_id, amount, qty, created_ts)
  VALUES (s.sale_id, s.amount, s.qty, CURRENT_TIMESTAMP);
-- CTE: Clean, readable, debuggable
WITH
staged AS (
  SELECT *, ROW_NUMBER() OVER (
    PARTITION BY order_id ORDER BY loaded_ts DESC
  ) AS rn
  FROM raw_orders
),
deduped AS (
  SELECT * FROM staged WHERE rn = 1
),
enriched AS (
  SELECT d.*, c.segment, p.category
  FROM deduped d
  LEFT JOIN dim_customer c ON d.customer_id = c.customer_id
  LEFT JOIN dim_product p ON d.product_id = p.product_id
)
SELECT segment, COUNT(*) AS orders, SUM(amount) AS revenue
FROM enriched
GROUP BY segment;

-- CTE vs temp table: CTE = in-memory, temp table = materialized
-- CTE: Fast to write, good for 1-2 levels of logic
-- Temp table: Better when downstream consumers query intermediate results
-- ❌ AVOID: Scanning more data than needed
SELECT * FROM fact_sales WHERE YEAR(order_date) = 2026;
-- ✅ PREFER: Filter on the clustering key directly
SELECT * FROM fact_sales WHERE order_date >= '2026-01-01' AND order_date < '2027-01-01';

-- ❌ AVOID: DISTINCT when GROUP BY is clearer
SELECT DISTINCT customer_id, COUNT(*) ...  -- wrong
-- ✅ PREFER
SELECT customer_id, COUNT(*) FROM ... GROUP BY customer_id;

-- ❌ AVOID: Cartesian join
SELECT * FROM big_table, another_big_table WHERE ...  -- missing JOIN condition
-- ✅ ALWAYS use explicit JOIN with ON condition

-- ❌ AVOID: Aggregating then joining
SELECT ... FROM (SELECT id, SUM(x) FROM big GROUP BY id) a JOIN dim b ON ...
-- ✅ PREFER: Filter/aggregate BEFORE join when possible
WITH small AS (SELECT id FROM dim WHERE active = TRUE)
SELECT ... FROM small s JOIN big b ON s.id = b.id ...

Quick checklist:

  • ✅ Filter early (WHERE before JOIN)
  • ✅ Aggregate before join
  • ✅ Use EXISTS instead of IN for subqueries on large tables
  • ✅ Use QUALIFY ROW_NUMBER() — Snowflake supports this directly
  • ❌ Don't use SELECT * in production — name columns

6 dbt — The Transformation Layer

dbt = నీ SQL transformations ని వెర్షన్ కంట్రోల్ లో పెట్టడం. ఇక్కడ నీకు మోడల్స్, రెఫ్స్, ఇంక్రిమెంటల్ బిల్డ్ — ఇవే ప్రధానం.
dbt = version control for your SQL transformations. Models, refs, incremental builds — these are the core concepts.

dbt Model Structure

-- models/staging/stg_orders.sql
-- Stage 1: Raw → Clean
WITH source AS (
  SELECT * FROM {{ source('ecommerce', 'raw_orders') }}
),
renamed AS (
  SELECT
    id AS order_id,
    customer_id,
    order_date::DATE AS order_date,
    amount::DECIMAL(10,2) AS amount,
    status
  FROM source
  WHERE status IS NOT NULL  -- data quality filter
)
SELECT * FROM renamed;

-- models/marts/dim_customer.sql
-- Stage 2: Clean → Dimension
{{ config(materialized='table', unique_key='customer_key') }}

SELECT
  customer_id,
  name,
  address,
  '2026-01-01'::DATE AS valid_from,
  NULL::DATE AS valid_to,
  TRUE AS is_current
FROM {{ ref('stg_customers') }};

-- models/marts/fact_orders.sql
-- Stage 3: Dimension + Fact
{{ config(materialized='incremental', unique_key='order_id') }}

SELECT
  o.order_id,
  d.date_key,
  c.customer_key,
  o.amount,
  o.status
FROM {{ ref('stg_orders') }} o
LEFT JOIN {{ ref('dim_date') }} d ON o.order_date = d.date
LEFT JOIN {{ ref('dim_customer') }} c ON o.customer_id = c.customer_id

{% if is_incremental() %}
  WHERE o.order_date > (SELECT MAX(order_date) FROM {{ this }})
{% endif %}

Key dbt commands you'll use daily:

dbt run        -- build all models (dbt resolves dependencies automatically)
dbt run -m stg_orders  -- build one model
dbt test       -- run schema tests (not_null, unique, accepted_values)
dbt docs generate && dbt docs serve  -- view lineage docs

8 Performance Tuning Quick Reference

SymptomDiagnosisFix
Query is slow
"taking forever"		 Check QUERY_PROFILE → look for:
• High % in "Scanning" → not pruning partitions
• High % in "Join" → no join key or skew
• Spilling to disk → warehouse too small 		• Scale up warehouse (XS → S → M)
• Add clustering key if scanning > 50%
• Filter early (WHERE before JOIN)
Dashboard is slow
"100 users all hitting refresh"		 Single warehouse + many concurrent queries = queue buildup Scale out: Multi-cluster warehouse (min 1, max 5). Each user group gets its own cluster.
ETL is slow
"data load taking hours"		 COPY INTO or dbt build using a small warehouse Scale up: Use a larger warehouse for the load (XL → 2XL). Suspend after done.
Same query keeps scanning same data
"no caching"		 Warehouse is suspending between runs, losing data cache Increase auto-suspend time. Or use result cache — same query within 24h is instant.

Snowflake Query Profile — How to Read

-- Get the query ID of a slow query
SELECT query_id, query_text, start_time, total_elapsed_time/1000 AS seconds
FROM TABLE(INFORMATION_SCHEMA.QUERY_HISTORY())
WHERE query_text LIKE '%your_table%'
ORDER BY start_time DESC
LIMIT 5;

-- Then inspect the profile
SELECT SYSTEM$EXPLAIN_QUERY_JSON('<query_id>');
-- Or in Snowsight: Query History → click query → Profile tab

-- What to look for in the profile:
-- • Bytes scanned (high = not enough partition pruning)
-- • Percentage spilled to local/remote (spilling = warehouse too small)
-- • Join type (PROBE JOIN vs HASH JOIN vs CARTESIAN — cartesian is bad)

9 Quick Check

1. What's the difference between scaling up and scaling out a Snowflake warehouse?
✅ Correct! Scale up = bigger compute per query (good for ETL). Scale out = more parallel clusters (good for BI dashboards with many users).
2. What grain should a fact_sales table have?
✅ Correct! Grain must be the most atomic level. If each "sale" has multiple items, grain = one row per sale_line_item. The grain defines what "one row" represents.
3. When a customer changes their address and you need to track the history, which SCD type do you use?
✅ Correct! Type 2 keeps full history. The old address is preserved in a previous row with valid_to set, and a new row with valid_from = today is added.
4. You have 50 analysts running dashboards. The warehouse is queuing queries. What's the fix?
✅ Correct! Queuing = concurrency problem = scale out (multi-cluster). Each cluster processes one query at a time, so 5 clusters handle 5 concurrent queries.
5. In dbt, which materialization should you use for a large fact table that gets new data daily?
✅ Correct! Incremental = only build what changed. For a 5B row fact table, full rebuild every day is insane. Incremental with unique_key and is_incremental() filter is the way.
