The top-level node — usually your company. Created when you sign up with a domain.

Folder

Groups projects by team, department, or environment (prod/staging/dev). Optional but recommended.

Project

The billing + IAM boundary. Every service lives inside exactly one project. Projects have an ID, name, and number.

Resources

The actual services — Cloud Run, BigQuery, Cloud Storage buckets, Compute Engine VMs. 150+ and growing.

7. Common Mistakes and How to Avoid Them

Missing EXPOSE / PORT

Cloud Run expects your app to listen on $PORT (default 8080). No EXPOSE 8080 in Dockerfile? Deployment fails.

Listening on wrong host

Must bind to 0.0.0.0, not 127.0.0.1. Otherwise Cloud Run's health checks can't reach your container.

Forgetting .dockerignore

Without it, the entire source directory gets sent to Cloud Build — including venv/, .git/, node_modules/. Slow builds and timeouts.

Billing disabled

Cloud Run needs billing enabled on the project — even for the free tier. If you skip this, deployment fails silently.

8. Interview Answers

30-second answer: What is Cloud Run?

"Cloud Run is a fully managed serverless compute platform on GCP that runs stateless containers. It auto-scales from zero to thousands of instances based on traffic. You only pay for CPU and memory during request processing — no requests, no cost. It's my go-to for deploying APIs and web backends."

90-second STAR: Deployment failure due to port mismatch

Situation: I deployed my first Flask app to Cloud Run. Task: Get it running with a public HTTPS endpoint. Action: I wrote the app, Dockerfile, ran gcloud run deploy. The deployment succeeded but health checks failed — the container kept crashing. I checked the logs: the app was listening on port 5000 (Flask default) instead of the $PORT env var Cloud Run injects (8080). Result: Fixed by reading $PORT from the environment: port = int(os.getenv("PORT", 8080)). Redeployed successfully. Impact: Learned the golden rule: Cloud Run injects the port — never hardcode it.

3-minute deep dive: How Cloud Run routing works

"When a request hits your Cloud Run service URL, the Cloud Run frontend (Google Front End / GFE) terminates TLS and routes to the service. It checks for an active revision — each deploy creates a new revision. If the revision has 0 instances, a cold start happens: a new container is pulled from Artifact Registry, your entrypoint runs, and once the container passes its startup health check, the request is forwarded. During the cold start, the request waits (up to the request timeout). After the response, the container stays warm for ~15 minutes of inactivity before being scaled to 0. For production, you can set a 'min instances' to keep 1+ always warm, at the cost of always-on pricing."
