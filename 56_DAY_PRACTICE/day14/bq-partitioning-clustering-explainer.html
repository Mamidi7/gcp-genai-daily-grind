What these do, in one line

Partitioning chops a table into date-based slices so queries skip entire irrelevant slices. Clustering sorts rows within each slice so block-level pruning works on non-date filters.

Unpartitioned scan reads all rows (left). Partition pruning reads 2 days (center). Clustering further prunes blocks within those 2 days by status (right).
Scanned Pruned Cluster-sorted

Block-by-block: How to use it

Create a partitioned table

CREATE TABLE `project.dataset.orders`
PARTITION BY DATE(order_ts)         ← partition column
CLUSTER BY status, region           ← sort within partition
OPTIONS(
  partition_expiration_days = 90,   ← auto-delete old partitions
  require_partition_filter = true   ← safety: force WHERE on partition
) AS
SELECT * FROM raw_orders;

Query that benefits from both

SELECT order_id, amount, status
FROM `project.dataset.orders`
WHERE order_ts >= '2025-04-01'      ← partition prune: reads 1 day → ~3% of data
  AND status = 'shipped'             ← cluster prune: scans blocks matching 'shipped'
  AND amount > 100;                  ← no index help: full filter on cluster-pruned set

Common Mistakes

Partitioning on a non-date column

You can only partition by DATE, TIMESTAMP, DATETIME, INT64 range, or ingestion time. Partitioning by STRING or FLOAT is not supported.

✓ Use DATE(transaction_ts) or ingestion_time-based partitioning

Too many partitions

BigQuery limits tables to 4,000 partitions. Daily partitioning with data older than 10 years will hit this limit. Old data gets rejected.

✓ Use monthly partitioning or set partition_expiration_days

Clustering on high-cardinality columns

Clustering sorts blocks. If a column has too many distinct values (like order_id), each block still contains most values — no pruning benefit.

✓ Cluster on columns with 10–500 distinct values (status, region, category)

Partitioning small tables

Tables under 10 GB get no benefit from partitioning. You pay for metadata ops without meaningful scan reduction.

✓ Only partition tables > 10 GB, or when partition_filter is a safety requirement

RULE Partition first

Date-based queries get 90%+ pruning with partitioning alone. Add clustering only when you also filter by a non-date column.

RULE Cluster columns: low cardinality

Status (shipped/pending/cancelled), region (US/EU/APAC), category. NOT order_id, user_id, or other high-cardinality fields.

RULE require_partition_filter = true

Forces every query to specify a partition filter. Prevents accidental full-table scans that cost thousands of dollars.

Interview Answers

30-second answer

What are partitioning and clustering in BigQuery?

"Partitioning divides a table into date-based slices so queries only scan relevant time ranges — reducing bytes and cost. Clustering sorts rows within each slice by specified columns so block-level pruning also works on non-date filters. Together they can reduce scanned bytes from 100% to under 2% for typical time-range queries with status filters."

3-minute technical walkthrough

How does BigQuery storage pruning work under the hood?

BigQuery uses a columnar storage format called Capacitor — similar to Parquet. Data is organized in storage blocks of ~100 MB, each storing metadata about the min/max values of sorted columns within that block.

When you run a query, the query planner reads the table's partition metadata first — a small table listing all partition boundaries. It eliminates partitions that don't match your WHERE clause. This is partition pruning.

Within surviving partitions, the planner reads cluster metadata — the min/max of each clustering column per block. If you query WHERE status = 'shipped' and blocks cluster rows by status, blocks whose min/max doesn't include 'shipped' are skipped entirely. This is cluster pruning.

The two form a cascade: partition pruning eliminates ~99% of storage at the date level, then cluster pruning eliminates another ~70% of remaining blocks at the filter level. The result: instead of scanning 100 blocks, you scan 1-2.

Key constraints: Clustering only works on the first 4 columns in cluster order. Clustering efficiency degrades with DML (INSERT/UPDATE/DELETE) because new rows may not be perfectly sorted — automatic re-clustering happens in the background but takes time. For real-time streaming tables, expect temporary cluster degradation.
