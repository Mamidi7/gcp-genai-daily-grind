## What Built

### 1. Input Validation with Pydantic Field Constraints
- `QuestionRequest` model with `@field_validator` and `Field()` constraints
- `question`: required, 1-500 chars, stripped length >= 3
- `category`: optional string
- Custom validator checks stripped length and raises `ValueError` if too short after trimming

### 2. Custom Exception Handler
- `request_validation_exception_handler` returns clean JSON:
  ```json
  {
    "error": "validation_failed",
    "message": "...",
    "fields": [{"type": "...", "loc": [...], "msg": "..."}]
  }
  ```
- `_sanitize()` function extracts only JSON-safe fields from `exc.errors()` — drops raw Python objects in `ctx` sub-dicts

### 3. Business Rule Exception (HTTPException)
- Reserved keyword check: questions starting with "ignore" (case-insensitive) return `400 Bad Request`
- Clean error response: `{"error": "reserved_keyword", "message": "..."}`

## Debug Cases Fixed

### Case 1: JSON Serialization Failure
- **Symptom**: `TypeError: Object of type ValueError is not JSON serializable` on blank body
- **Root cause**: `exc.errors()` returns Pydantic error dicts with raw Python objects inside `ctx` sub-dicts
- **Fix**: `_sanitize()` strips to only `type`, `loc`, `msg` fields
- **Prevention**: Never pass Pydantic error objects directly to JSON. Always extract safe string fields.

### Case 2: Response Model Type Mismatch
- **Symptom**: `ResponseValidationError: 'clean_char_count' must be a string`
- **Root cause**: Return annotation `-> dict[str, str]` told FastAPI to enforce string values for ALL dict entries
- **Fix**: Removed restrictive return type annotation
- **Prevention**: Return type hints in FastAPI trigger response validation. Omit when returning mixed types.

### Case 3: Case Sensitivity in Business Rule
- **Symptom**: "ignore this" caught but "IGNORE this" was not
- **Fix**: `payload.question.lower().startswith("ignore")`
