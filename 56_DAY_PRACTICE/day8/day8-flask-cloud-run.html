Concept Explainer

VMs vs Containers vs Cloud Run — Transportation Logic lo
🚗 Own Car vs Zoomcar vs Uber

VM (సొంత కారు / Own Car): You buy the car. Maintenance, insurance, parking — all on you. Full control, full responsibility. Just like a VM where you manage the OS, security patches, and the app.

Docker Container (జూమ్ కార్ / Zoomcar): Rent a self-drive car. No worries about long-term maintenance. You bring your skills and go. Portable, but you still pay for the time you hold onto it.

Cloud Run (ఓలా / Uber): Just book a ride. You don't drive, you don't park. Get in, reach destination, pay only for the distance. Ride over? Zero cost. That's Cloud Run scale-to-zero.

Cold Start — Geyser Logic

Another way to think about it
🥶 The Geyser Analogy

Imagine you want to take a hot shower in the morning.

If you turn on the geyser only when you step into the bathroom, you have to wait 5-10 minutes for the water to heat up. That waiting time? That is your cold start latency.

But what if you leave the geyser ON 24/7? The water is instantly hot whenever you want it! The electricity bill keeps running though.

❄️
Geyser OFF = min-instances=0

Scale-to-zero. Cheap when idle. But first request takes 100ms - 3s to pull image + start container.

🔥
Geyser ON = min-instances=1

Always warm. Instant response. But you pay a little more to keep that instance running 24/7.

Architecture — 4 Blocks, 1 Pipeline

Browser nundi Flask varaku request ela travel chestundi — Block by block.
Telugu lo — Full Pipeline

Browser: You open the app on your phone and send a request. "Data kavali" — the browser leads the request, first in line.

Cloud Run (Gate keeper): Your request arrives. Cloud Run checks it — "sarigga undi, security clear" — then routes it to the right worker. Reverse proxy, autoscaling, HTTPS — it handles everything.

Gunicorn (Manager): 4-5 workers ready. The one with the lightest load gets your request. One worker busy? Another picks it up. Concurrency handled.

Flask App (Chef): Request lands on the right route. /health? Returns "ok". /? Returns "Hello". Pure application logic.

Flow: You → Cloud Run (gate) → Gunicorn (manager) → Flask (chef) → Response back

main.py — Avoid vs Prefer

Dev pattern vs Production pattern — side by side comparison.
❌
Avoid This
# Dev-only — don't deploy this! from flask import Flask app = Flask(__name__) @app.route('/') def hello(): return "Hello" # Issues: # ① No /health endpoint # ② Port hardcoded (5000) # ③ No gunicorn app.run(host='127.0.0.1', port=5000)

✗ No health check → Cloud Run can't verify readiness
✗ Port hardcoded → PORT env mismatch
✗ host='127.0.0.1' → unreachable in container

✅
Production Pattern
# Production-ready from flask import Flask, jsonify import os app = Flask(__name__) @app.route('/') def root(): return jsonify({"message": "Production ready!"}) # ① Health endpoint @app.route('/health') def health(): return jsonify({"status":"healthy"}) # ② if __name__ guard if __name__ == "__main__": # ③ PORT from env port = int(os.environ.get("PORT", 8080)) # ④ host='0.0.0.0' app.run(host="0.0.0.0", port=port)

✓ /health endpoint for readiness
✓ PORT from env — Cloud Run compatible
✓ gunicorn ready — concurrency handled

Docker & Dependencies

Container recipe — requirements.txt, Dockerfile, .dockerignore
Why gunicorn matters

Flask's dev server is single-threaded. Oka time lo oka request matrame handle chestundi. Cloud Run sends concurrent requests! Without gunicorn, requests queue up, the container looks dead, and Cloud Run restarts it. Gunicorn forks multiple workers to handle the load cleanly.

flask>=2.0.0 gunicorn>=20.0.0
# ── Layer 1: Base image (minimal OS) ── FROM python:3.11-slim WORKDIR /app # ── Layer 2: Dependencies FIRST (layer caching ⚡) ── COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt # ⚡ Key: This CACHES! Code changes ≠ pip reinstall. # Build takes ~5s instead of ~60s. # ── Layer 3: Application code ── COPY . . EXPOSE 8080 # ── Layer 4: Production entry point ── CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"] # Exec form (JSON array) — forwards SIGTERM correctly
⚡
Layer Caching

requirements.txt ni code BEFORE copy chestam — Docker layer cache work avutundi. Code-only changes = 5s rebuild.

Deploy & Verify

Raa bro — ee steps exactly follow cheyyi. Complete each step in Cloud Shell.

GCP Console Task

4 steps — Enable APIs → Build → Deploy → Verify
  1. Enable required APIs
    gcloud services enable run.googleapis.com cloudbuild.googleapis.com
  2. Build and push Docker image
    gcloud builds submit --tag gcr.io/$GOOGLE_CLOUD_PROJECT/flask-app:v1 .
  3. Deploy to Cloud Run
    gcloud run deploy flask-app \ --image gcr.io/$GOOGLE_CLOUD_PROJECT/flask-app:v1 \ --platform managed \ --region us-central1 \ --allow-unauthenticated
  4. Test the live endpoint
    curl https://YOUR_CLOUD_RUN_URL/health # Expected: {"status": "healthy"}
✅ Success looks like:
  • Cloud Run URL active and returning JSON.
  • Cloud Run console shows "1 revision serving 100% traffic".
  • Dashboard shows a green checkmark for health checks.
