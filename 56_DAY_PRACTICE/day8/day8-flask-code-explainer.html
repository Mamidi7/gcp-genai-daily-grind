# 1. Import Flask
from flask import Flask, jsonify
import os

# 2. Create the Flask app
app = Flask(__name__)

# 3. Define routes
@app.route('/')
def hello():
    return jsonify({
        "message": "Hello from Cloud Run!",
        "status": "deployed",
        "platform": "Google Cloud Run"
    })

@app.route('/health')
def health():
    return jsonify({"status": "healthy"})

@app.route('/info')
def info():
    return jsonify({
        "app": "Flask on Cloud Run",
        "port": os.environ.get("PORT", 8080)
    })

# 4. Entry point — only runs when executed directly
if __name__ == "__main__":
    port = int(os.environ.get("PORT", 8080))
    app.run(host="0.0.0.0", port=port)
1
from flask import Flask, jsonify — Flask is a lightweight Python web framework. `jsonify` converts Python dicts into JSON responses. import os reads environment variables (like PORT).
2
app = Flask(__name__) — Creates the Flask application. __name__ tells Flask where to look for templates/static files. This app object handles all incoming HTTP requests.
3
@app.route() — A Python decorator. It tells Flask: "When someone visits this URL path, run the function below." Each decorated function returns a JSON response via jsonify().
4
if __name__ == "__main__" — Only runs when you execute python main.py directly. When gunicorn runs this (in production), this block is ignored. host="0.0.0.0" means "accept connections from anywhere" (required inside a container). PORT defaults to 8080 — Cloud Run's expected port.

requirements.txt — Dependencies

flask>=2.0.0
gunicorn>=20.0.0
1
flask — The web framework. >=2.0.0 means "any version 2.0 or newer". We use this to build the API.
2
gunicorn — A production WSGI server. Flask's built-in server is single-threaded (good for dev). Gunicorn handles multiple requests in parallel. Cloud Run uses it to serve traffic properly.

🔴 Forgetting gunicorn

If you deploy with only flask (no gunicorn), Cloud Run will timeout or crash under load because Flask's dev server can't handle concurrent requests.

✓ Always include gunicorn in requirements.txt for production

🔴 Using flask run directly

CMD ["python", "main.py"] in Dockerfile uses Flask dev server — fine for testing, not for production traffic.

✓ Use CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"]

Dockerfile — The Container Recipe

# 1. Base image
FROM python:3.11-slim

# 2. Working directory inside container
WORKDIR /app

# 3. Copy dependencies first (layer caching)
COPY requirements.txt .

# 4. Install Python packages
RUN pip install --no-cache-dir -r requirements.txt

# 5. Copy the rest of the code
COPY . .

# 6. Document the port
EXPOSE 8080

# 7. Start gunicorn
CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"]
1
FROM python:3.11-slim — Starts from an official Python image. Slim = minimal OS layer (smaller image size, fewer security holes). Full image would be 1GB+, slim is ~120MB.
2
WORKDIR /app — All subsequent commands run from this directory inside the container. Like cd /app.
3
COPY requirements.txt . — Copy only the requirements file first. This is a Docker caching trick: if requirements.txt hasn't changed, Docker reuses the cached pip install layer instead of reinstalling everything.
4
COPY . . — Copy the actual app code (main.py). This happens AFTER pip install so code changes don't trigger a full reinstall.
5
CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"] — The command that runs when the container starts. main:app means "from main.py, use the variable named 'app'". --bind 0.0.0.0:8080 tells gunicorn to listen on port 8080 from all network interfaces.

🔴 Wrong CMD syntax

CMD ["python", "main.py"] instead of gunicorn — Flask dev server can't handle multiple requests. The app will timeout under load.

✓ CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"]

🔴 Not using .dockerignore

Without it, your local venv, __pycache__, .git all get copied into the image, making it huge and slow to upload.

✓ Add .dockerignore with __pycache__/, .git, *.pyc, venv/

🔴 Hardcoding port

Cloud Run sets the PORT env var dynamically. If you hardcode port 5000 or 80, Cloud Run can't route traffic to you.

✓ Always use os.environ.get("PORT", 8080) and match EXPOSE + CMD

How the 3 Files Connect

Flow: requirements.txt lists what's needed → Dockerfile builds the container → Cloud Build executes the build → Container runs on Cloud Run

Common Mistakes

🔄 PORT mismatch

Dockerfile EXPOSEs 8080, CMD binds 8080, but the app uses os.getenv("PORT", 5000). These must all agree.

✓ Keep PORT=8080 as default everywhere

📦 requirements.txt missing

pip install -r requirements.txt fails if the file isn't copied first. Docker layer caching makes this a two-step COPY.

✓ COPY requirements.txt . before RUN pip install

🌐 0.0.0.0 vs 127.0.0.1

127.0.0.1 only allows connections from inside the container. Cloud Run's proxy connects externally → needs 0.0.0.0.

✓ app.run(host="0.0.0.0", port=port)

🚫 @app.route indentation

If the function body isn't indented properly under the decorator, Flask won't register the route.

✓ Keep 4-space indentation consistently

🔧 3-Minute Technical Walkthrough

Architecture: The request lifecycle is: Internet → Cloud Run proxy → container's gunicorn → Flask router → handler function → JSON response. Cloud Run automatically scales the container from 0 to N instances based on incoming traffic.

Key Design Decisions:

  • Slim base image — python:3.11-slim over full: 120MB vs 1GB+ image size, faster cold starts
  • gunicorn over Flask dev server — Dev server is single-threaded, gunicorn forks workers to handle concurrent requests. Without this, multiple users would block each other.
  • Layer caching — COPY requirements.txt first means code changes don't reinstall packages. Builds go from 60s → 5s on repeat deploys.
  • Health check endpoint — Cloud Run's infrastructure polls /health every few seconds. If it returns non-200, the instance is killed and restarted.

Failure modes:

  • If gunicorn isn't installed, the container starts but crashes immediately (ModuleNotFoundError)
  • If PORT env doesn't match between Dockerfile and app, Cloud Run's health check fails and never routes traffic
  • If the image is too large (full python image), cold starts take 10+ seconds versus 2-3 seconds for slim
