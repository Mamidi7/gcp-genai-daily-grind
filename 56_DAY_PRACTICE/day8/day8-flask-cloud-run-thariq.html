Telugu lo — Full Stack

Telugu lo — Request Lifecycle

Browser = Nuvvu mobile pattukuni app open chesav la. "Data kavali" ani request pampistav. Browser ki vere pani ledu — adi request ni Cloud Run ki forward chestadi. Idantha browser though start avutundi.

Cloud Run = Gate keep la untadi. Nee request ostundi, adi chusukuni — "sarigga undi, security clear" ani verify chestundi. Reverse proxy, HTTPS termination, autoscaling — idantha Cloud Run automatic ga chestadi. Nee code ki direct ga request raakunda, Cloud Run madhya lo filter la untadi.

Gunicorn = Manager la untadi. Nee request wochindi. Gunicorn daggara ready ga 4-5 workers untaru (default). Evariki pani takkuvo vallaki ee request assign chestadu. Concurrency handle cheyadaniki — okadu busy ante inko worker untadi. Flask dev server single-threaded — adi handle cheyaledu, but gunicorn cheyagaladu.

Flask App = Chef la untadi. Request vachindi, specific route batti logic run chestadu. "/health" ante "ok" returns, "/" ante metadata JSON returns. Pure application logic — input thisko, output ivvali.

Problem solve chestundi: Without gunicorn, Flask handles one request at a time. Under load, requests queue up, timeouts occur, Cloud Run kills the instance. With gunicorn, concurrent requests are handled by separate workers — no timeout.

Problem kuda undi: Gunicorn sync workers block on I/O. For long database queries or external API calls, all workers can block. Async workers (uvicorn) or async routes solve that — but that's a different stack.

Flow: Nuvvu → Cloud Run (gate) → Gunicorn (manager) → Flask (chef) → Response back → Nuvvu chuskovadam

3 Files, Block by Block

Every file explained: what it does, why it's there, what goes wrong without it.

main.py — Application Logic

Application Logic

Three endpoints: / returns metadata, /health for liveness probes, /info with dynamic config. Uses jsonify(). The if __name__ == "__main__" block runs only in dev — gunicorn imports the module and this block is skipped.

# 1. Imports — Flask and jsonify for JSON responses from flask import Flask, jsonify import os # 2. App instance — single Flask application app = Flask(__name__) # 3. Routes — / for metadata, /health for probes, /info for config @app.route('/') def hello(): return jsonify({ "message": "Hello from Cloud Run!", "status": "deployed", "platform": "Google Cloud Run" }) @app.route('/health') def health(): return jsonify({"status": "healthy"}) @app.route('/info') def info(): return jsonify({ "app": "Flask on Cloud Run", "port": os.environ.get("PORT", 8080) }) # 4. Dev entry point — skipped when gunicorn imports this module if __name__ == "__main__": port = int(os.environ.get("PORT", 8080)) app.run(host="0.0.0.0", port=port)
! Pattern: os.environ.get("PORT", 8080) — Cloud Run sets $PORT dynamically. Must match app.run(), EXPOSE, and gunicorn --bind. If they disagree, the container starts but Cloud Run health checks fail.

requirements.txt — Dependencies

Two Packages

Flask (routing) + gunicorn (production server). Flask dev server handles one request at a time — gunicorn forks multiple workers for concurrency.

flask>=2.0.0 gunicorn>=20.0.0
Common Mistake: Deploying without gunicorn. Flask's dev server is single-threaded. Under Cloud Run's load balancer, concurrent requests queue up, time out, and Cloud Run kills the instance as unhealthy.

Dockerfile — Container Recipe

Layer Caching Strategy

Base image python:3.11-slim (~120MB vs 1GB+ full). Dependencies install BEFORE code copy — code-only changes rebuild in seconds (layer cache hit).

# 1. Base image — minimal OS layer FROM python:3.11-slim # 2. Working directory — inside the container WORKDIR /app # 3. Copy deps first — layer caching means this step is cached until requirements.txt changes COPY requirements.txt . # 4. Install — cached until step 3 changes RUN pip install --no-cache-dir -r requirements.txt # 5. Copy the app code — this changes every deploy COPY . . # 6. Document the port (documentation only — Cloud Run sets the actual port) EXPOSE 8080 # 7. Run — gunicorn with 4 workers, binding to $PORT CMD ["gunicorn", "-w", "4", "-b", "0.0.0.0:8080", "app:app"]
! Order matters: Line 3-4 (copy deps + install) before line 5 (copy code). If deps haven't changed, Docker uses the cached layer. If you copy code first, every deploy re-installs everything — 2-3 minutes extra per deploy.

Deploy Steps

1 · Build and test locally ~1 min

Run the Flask dev server to verify routes work:

python main.py # Visit http://localhost:8080, /health, /info

Test each endpoint with curl or browser. Check that JSON responses are correct. If routes fail locally, they'll fail on Cloud Run too — and logs are harder to read on Cloud Run.

2 · Docker build and run locally ~2 min

Containerize and verify the Docker image works:

docker build -t flask-app . docker run -p 8080:8080 flask-app

This catches Dockerfile bugs before deployment — wrong base image, missing dependencies, wrong CMD syntax.

3 · Deploy to Cloud Run ~3 min

One command builds, pushes to Artifact Registry, and deploys:

gcloud run deploy flask-app \ --source . \ --region us-central1 \ --allow-unauthenticated

Cloud Build reads the Dockerfile, installs deps, copies code, pushes the image, and deploys. Output includes the live URL (ending in .run.app).

4 · Verify production ~30s

Test the live Cloud Run URL:

curl https://flask-app-xxxxx-uc.a.run.app/ curl https://flask-app-xxxxx-uc.a.run.app/health curl https://flask-app-xxxxx-uc.a.run.app/info

Check Cloud Run dashboard for request count, latency, and errors. Verify no cold start issues by hitting the URL after 15 minutes of idle.

Interview Prep

Three depths — pick the one that fits the question.

30-Second Answer

Flask + Docker + Cloud Run is a 3-file pattern: main.py defines the API routes, requirements.txt lists Flask and Gunicorn, and Dockerfile packages everything in a Python 3.11-slim image. A single gcloud run deploy --source . builds, pushes, and deploys to Cloud Run. Gunicorn provides multi-worker concurrency (Flask dev server is single-threaded). Cloud Run auto-scales to zero when idle, charges only for request time.

90-Second STAR Answer

Situation: Needed to deploy a lightweight API service that could scale from zero to handle traffic spikes without managing servers.

Task: Build a containerized Python web service deployable with a single command, minimizing infrastructure overhead and cold-start latency.

Action: Wrote a Flask app with three JSON endpoints (/, /health, /info). Used python:3.11-slim base image (~120MB vs 1GB+). Structured the Dockerfile with dependency caching (copy requirements.txt before code). Deployed with gcloud run deploy --source .. Configured Gunicorn with 4 workers for concurrency.

Result: Deployed in under 3 minutes. Cold start ~1-2 seconds. Handles 100+ concurrent requests with zero provisioning. Reduced infrastructure management to zero — no VMs, no Kubernetes, no load balancers.

Insight: The trick is layer caching in the Dockerfile. Copying deps before code means 90% of builds skip the pip install step — deploys go from 3 minutes to 30 seconds.

3-Minute Technical Walkthrough

Start with the 3-file structure. main.py defines Flask routes returning jsonify() — the if __name__ == "__main__" guard ensures gunicorn imports the module without starting the dev server. The $PORT environment variable must match across app.run(), EXPOSE, and gunicorn --bind — Cloud Run injects it dynamically, and if they disagree health checks fail.

The Dockerfile uses python:3.11-slim for minimal image size (fewer CVEs, faster pulls). Layer ordering is deliberate: COPY requirements.txt before code ensures pip install is cached across code-only rebuilds. Each layer change triggers a rebuild of all subsequent layers — so the most volatile layers (code) come last.

Gunicorn with 4 workers handles 4 concurrent requests. Flask dev server is single-threaded (1 at a time). Under Cloud Run's load balancer, concurrent requests queue up in the dev server, timeout, and Cloud Run marks the instance unhealthy. Gunicorn's pre-fork model spawns separate OS processes — each worker blocks independently on I/O (database, external API).

Limitations: Sync workers block on I/O. For 100+ concurrent requests needing DB queries, switch to async workers (uvicorn) or increase workers based on instance memory. Cold start is ~1-2s on first request after idle; use min-instances to keep one warm for zero cold start.

One-liner to remember: gcloud run deploy --source . = build + push + deploy. No CI/CD needed for prototypes.
