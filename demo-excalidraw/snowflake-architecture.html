The 3 Layers Explained

What each layer does and why it matters for interviews

1. Cloud Services Layer

The brain of Snowflake. Global, stateless, always running. Handles authentication, query optimization, metadata management, MVCC transactions, and the Result Cache (24-hour storage of query results). You never pay for this layer — it's included in Snowflake's pricing.

Interview 30s Answer

Q: "How does Snowflake optimize queries without compute?" — Cloud Services is a global stateless layer handling auth, parsing, optimization, and metadata. The optimizer reads micro-partition metadata to prune files. Zero compute credits for planning.

2. Compute Layer (Virtual Warehouse)

The muscle. VWs are elastic compute clusters (X-Small to 6X-Large). Multi-cluster auto-scaling for concurrency. Local SSD cache for hot data. Auto-suspend on idle (default 10min, zero cost when suspended). This is what you pay for — per-second billing while running.

Common Confusion

"Does the VW store data permanently?" — No. Local SSD cache helps hot queries, but the source of truth is always cloud blob storage. Suspend one VW, resume another — it reads from storage fresh.

3. Storage Layer

The memory. Data stored in cloud blob store (S3/GCS/Azure Blob) in compressed columnar format. Micro-partitions (50-500MB, self-contained, immutable). Clustering defines sort order for pruning. Time Travel (1-90 days of point-in-time queries). Fail-safe (7 days, Snowflake-managed).

SQL Walkthrough

Key Snowflake SQL patterns — staging to query

1. Create Warehouse + Database

CREATE WAREHOUSE my_wh WAREHOUSE_SIZE = 'XSMALL' AUTO_SUSPEND = 300 AUTO_RESUME = TRUE INITIALLY_SUSPENDED = TRUE; CREATE DATABASE analytics_db; CREATE SCHEMA staging;

2. Stage + File Format

CREATE FILE FORMAT csv_format TYPE = 'CSV' FIELD_DELIMITER = '|' SKIP_HEADER = 1; CREATE STAGE my_stage URL = 's3://my-bucket/data/' FILE_FORMAT = csv_format;

3. COPY INTO

COPY INTO analytics_db.staging.orders FROM @my_stage/orders/ FILE_FORMAT = csv_format ON_ERROR = 'CONTINUE' PURGE = TRUE;
ON_ERROR = 'CONTINUE' is production-critical. Without it, one bad row fails the entire batch. With it, bad rows are skipped — inspect via VALIDATION_MODE.
