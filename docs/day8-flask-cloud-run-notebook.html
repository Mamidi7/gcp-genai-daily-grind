The Code

Four core files with inline explanations and copy support.

# Flask app with 3 routes from flask import Flask, jsonify import os app = Flask(__name__) @app.route('/') def home(): return jsonify({ "message": "Hello from Cloud Run!", "status": "deployed", "platform": "Google Cloud Run" }) @app.route('/health') def health(): return jsonify({"status":"healthy"}) @app.route('/info') def info(): return jsonify({ "app":"Flask on Cloud Run", "port": os.environ.get("PORT", 8080) }) if __name__ == "__main__": app.run(host="0.0.0.0", port=int(os.environ.get("PORT", 8080)))
Pattern
if __name__ == "__main__" runs only during local execution; Gunicorn imports the module in prod.
flask>=2.0.0 gunicorn>=20.0.0
Why gunicorn
Cloud Run can route concurrent requests. Gunicorn gives worker concurrency; Flask dev server is not for production traffic.
FROM python:3.11-slim WORKDIR /app COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt COPY . . EXPOSE 8080 CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"]
Layer caching
Copy requirements before app code to avoid reinstalling dependencies on every code edit.
__pycache__/ *.pyc .venv/ venv/ .git/ .env .DS_Store *.egg-info/

Common Mistakes

Six mistakes that repeatedly break Flask-on-Cloud-Run deployments.

1
Missing gunicorn in prod CMD.
2
PORT mismatch across app and runtime.
3
Binding to 127.0.0.1 instead of 0.0.0.0.
4
No .dockerignore; massive slow context.
5
Wrong Docker layer order, cache misses.
6
Shell CMD traps signals unexpectedly.
