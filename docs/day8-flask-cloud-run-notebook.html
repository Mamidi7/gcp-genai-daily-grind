The Code — Block by Block

Every file and every line explained: what it does, why it's there.
Application Logic

main.py — 3 routes, 1 Flask app, JSON responses

Three endpoints: / (hello message with service metadata), /health (Cloud Run health check — always 200), /info (dynamic config like PORT). Uses jsonify() for structured JSON. The if __name__ == "__main__" block only runs in dev.
# ===== 1. Imports ===== from flask import Flask, jsonify import os # ===== 2. App instance ===== app = Flask(__name__) # ===== 3. Routes ===== @app.route('/') def hello(): return jsonify({ "message": "Hello from Cloud Run!", "status": "deployed", "platform": "Google Cloud Run" }) @app.route('/health') def health(): return jsonify({"status": "healthy"}) @app.route('/info') def info(): return jsonify({ "app": "Flask on Cloud Run", "port": os.environ.get("PORT", 8080) }) # ===== 4. Dev entry point ===== if __name__ == "__main__": port = int(os.environ.get("PORT", 8080)) app.run(host="0.0.0.0", port=port)
🔑 Key Pattern: if __name__ == "__main__"
This block only runs when you execute python main.py directly. When gunicorn imports this module in production, this block is skipped. That's why the Dockerfile uses CMD gunicorn ... instead.
🔑 Key Pattern: os.environ.get("PORT", 8080)
Cloud Run sets the PORT env var dynamically. Never hardcode — always read from env with a sensible default (8080). The port must match between app.run(), EXPOSE, and gunicorn --bind.
Dependencies

requirements.txt — Exactly 2 packages

flask for the web framework and gunicorn for production serving. Flask's dev server (app.run()) is single-threaded — gunicorn spawns multiple worker processes to handle concurrent requests.
flask>=2.0.0 gunicorn>=20.0.0
🐍
flask

The micro web framework. >=2.0.0 ensures we get async-compatible version. Size: ~200KB.

⚡
gunicorn

Production WSGI server. Fork-based worker model handles concurrent requests. Without it, one slow request blocks all others.

⚠️ Common Mistake: Deploying without gunicorn. Flask's dev server (python main.py) can only handle one request at a time. Under Cloud Run's load balancer, the instance will timeout and restart in a loop. Always use gunicorn for production.
Container Recipe

Dockerfile — 7 lines that build the image

Uses python:3.11-slim as base (~120MB vs 1GB+ for full). Layer caching: requirements.txt is copied and installed before the code, so code changes don't trigger a full pip reinstall.
# 1. Base image — minimal OS layer FROM python:3.11-slim # 2. Working directory inside container WORKDIR /app # 3. Copy deps first (layer caching) COPY requirements.txt . # 4. Install — cached until requirements changes RUN pip install --no-cache-dir -r requirements.txt # 5. Copy the app code (changes often) COPY . . # 6. Document the port EXPOSE 8080 # 7. Production entry point CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"]
🔑 Layer Caching
Every COPY and RUN creates a Docker layer. Layers are cached — if the file hasn't changed, Docker reuses the cached layer. By copying requirements.txt before the code, we ensure pip install only re-runs when dependencies change. Code-only updates rebuild in seconds instead of minutes.
🔑 CMD format
CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"] — JSON array format (exec form), not shell form (CMD gunicorn ...). Exec form sends signals (SIGTERM for graceful shutdown) correctly. Shell form spawns a sh -c wrapper that traps signals.
Clean Builds

.dockerignore — What NOT to copy

Without .dockerignore, your venv, __pycache__, .git, and local temp files all get copied into the image. This makes builds slow and images bloated. Always exclude dev artifacts.
__pycache__/ *.pyc .env .git .gitignore venv/ .venv/ .DS_Store *.egg-info/
💡 Impact: Without .dockerignore, a 5MB Flask app can balloon to 200MB+ in the build context. Cloud Build uploads the entire context — .dockerignore keeps it lean.

Common Mistakes + Fixes

Real errors from real Cloud Run deployments — symptom, root cause, fix.
1
Missing gunicorn

Symptom: Container starts but crashes under load. Root cause: CMD uses python main.py — Flask dev server can't handle concurrent requests. Fix: CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"]

2
PORT mismatch

Symptom: Health checks fail, container restarts in a loop. Root cause: Cloud Run sets PORT=8080 but app listens on 5000. Fix: os.environ.get("PORT", 8080) everywhere — main.py, EXPOSE, CMD.

3
No .dockerignore

Symptom: Build takes 2+ minutes, image is 500MB+. Root cause: virtualenv, __pycache__, .git all copied into context. Fix: Add .dockerignore excluding dev artifacts.

4
127.0.0.1 vs 0.0.0.0

Symptom: App works locally but fails on Cloud Run. Root cause: app.run(host="127.0.0.1") only accepts local connections inside the container. Cloud Run's proxy connects externally. Fix: host="0.0.0.0"

5
Wrong CMD format

Symptom: Container exits immediately. Root cause: Shell form CMD gunicorn ... doesn't forward signals. Fix: Exec form: CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"]

6
Layer order wrong

Symptom: Code change triggers full pip reinstall (2+ min build). Root cause: COPY . . before RUN pip install. Fix: COPY requirements.txt first → RUN pip → then COPY app code.

Deployment — Local → Cloud Run

From python main.py to a live HTTPS endpoint.
Figure: 4 stages — Dev → Docker → Registry → Cloud Run
1
Local test

python main.py — test at http://localhost:8080 with curl or browser

2
Containerize

gcloud builds submit --tag gcr.io/$PROJECT/flask-app — Cloud Build triggers on the Dockerfile

3
Deploy

gcloud run deploy flask-app --image gcr.io/$PROJECT/flask-app --region us-central1 --allow-unauthenticated

4
Verify

curl https://flask-app-xxxxx-uc.a.run.app/ — should return JSON with message + status

⚡ One-command shortcut: gcloud run deploy --source . --region us-central1 --allow-unauthenticated combines build + push + deploy in one step. Cloud Build detects the Dockerfile, builds the image, pushes to Artifact Registry, and deploys to Cloud Run.

⚡ 30-Second Answer
I built a Flask web app with 3 JSON endpoints (/, /health, /info), packaged it with gunicorn in a Docker container using python:3.11-slim, and deployed it to Cloud Run. The Dockerfile uses layer caching — dependencies install first, code copies second. Cloud Run handles HTTPS, auto-scaling, and health checks via the /health endpoint.
🎯 90-Second STAR Answer
S: I needed to build and deploy a lightweight API service on Google Cloud Run for a serverless project.
T: The service needed health checks for Cloud Run's load balancer, containerized for portability, and able to handle concurrent requests.
A: I built a Flask app with 3 routes using jsonify() for structured JSON. I added gunicorn in requirements.txt for production concurrency. The Dockerfile uses python:3.11-slim (120MB vs 1GB+) with layer caching — COPY requirements.txt first, RUN pip install, then COPY the app code. For deployment, I used gcloud run deploy --source . which triggers Cloud Build to containerize and deploy in one command.
R: The service deployed successfully at a *.run.app URL with all 3 endpoints responding. Build time is ~60s for first deploy and ~5s for subsequent code-only updates thanks to layer caching. The /health endpoint keeps the service alive under Cloud Run's auto-healing.
🔧 3-Minute Technical Walkthrough
Architecture: The request lifecycle is: Internet → Cloud Run HTTPS proxy → container → gunicorn (master process) → gunicorn (worker process) → Flask router → handler function → JSON response. Cloud Run scales from 0 to N instances based on traffic volume.

Key Decisions:
  • Slim base image: python:3.11-slim over full: 120MB vs 1GB+. Faster cold starts (2-3s vs 10+ seconds). Smaller attack surface.
  • Gunicorn over Flask dev server: Dev server is single-threaded — it handles one request at a time. Gunicorn forks multiple worker processes. Without this, concurrent users would block each other.
  • Layer caching: Docker caches each layer. By copying requirements.txt first and installing dependencies before copying the app code, we ensure code-only changes don't trigger a full pip reinstall. Builds go from ~60s to ~5s.
  • Health check endpoint: Cloud Run's infrastructure polls /health every few seconds. If it returns non-200, the instance is killed and restarted. This provides auto-healing without extra config.
Failure modes:
  • Missing gunicorn → container starts but crashes under concurrent load
  • PORT mismatch → health check fails → endless restart loop
  • host="127.0.0.1" → works locally but Cloud Run proxy can't connect
  • No .dockerignore → venv+git copied → slow builds, large images
