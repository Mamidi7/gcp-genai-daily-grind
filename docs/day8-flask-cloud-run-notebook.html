System Overview

3 files with distinct jobs: main.py (app logic — 3 routes returning JSON), requirements.txt (dependencies — flask + gunicorn), and Dockerfile (container recipe — slim Python image, layer caching, CMD). Deployed to Cloud Run which auto-scales and manages HTTPS.

The Code — Block by Block

Every file and every line explained: what it does, why it's there.
Application Logic

Three endpoints: / (hello with service metadata), /health (Cloud Run health check), /info (dynamic config). Uses jsonify() for structured JSON. The if __name__ == "__main__" block runs only in dev.

# ===== 1. Imports ===== from flask import Flask, jsonify import os # ===== 2. App instance ===== app = Flask(__name__) # ===== 3. Routes ===== @app.route('/') def hello(): return jsonify({ "message": "Hello from Cloud Run!", "status": "deployed", "platform": "Google Cloud Run" }) @app.route('/health') def health(): return jsonify({"status": "healthy"}) @app.route('/info') def info(): return jsonify({ "app": "Flask on Cloud Run", "port": os.environ.get("PORT", 8080) }) # ===== 4. Dev entry point ===== if __name__ == "__main__": port = int(os.environ.get("PORT", 8080)) app.run(host="0.0.0.0", port=port)
Pattern: if __name__ == "__main__"
This block runs only when you execute python main.py directly. When gunicorn imports this module in production, this block is skipped.
Pattern: os.environ.get("PORT", 8080)
Cloud Run sets the PORT env var dynamically. The port must match between app.run(), EXPOSE, and gunicorn --bind.
Dependencies

flask for the web framework and gunicorn for production serving. Flask's dev server is single-threaded — gunicorn spawns multiple worker processes for concurrent requests.

flask>=2.0.0 gunicorn>=20.0.0
F
flask

The micro web framework. >=2.0.0 ensures async compatibility. Size: ~200KB.

G
gunicorn

Production WSGI server. Fork-based worker model handles concurrent requests.

Common Mistake: Deploying without gunicorn. Flask's dev server handles one request at a time — under Cloud Run's load balancer, the instance will timeout and restart.
Container Recipe

Uses python:3.11-slim base (~120MB vs 1GB+ for full). Layer caching: requirements.txt is copied before the code, so code changes don't trigger a full pip reinstall.

# 1. Base image — minimal OS layer FROM python:3.11-slim # 2. Working directory WORKDIR /app # 3. Copy deps first (layer caching) COPY requirements.txt . # 4. Install — cached until requirements changes RUN pip install --no-cache-dir -r requirements.txt # 5. Copy the app code COPY . . # 6. Document the port EXPOSE 8080 # 7. Production entry point CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"]
Layer Caching
Every COPY and RUN creates a Docker layer. By copying requirements.txt before the code, pip install only re-runs when dependencies change. Code-only updates rebuild in seconds instead of minutes.
CMD Format
Exec form (CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"]) sends signals (SIGTERM) correctly. Shell form (CMD gunicorn ...) wraps in sh -c which traps signals.
Clean Builds

Without .dockerignore, venv, __pycache__, .git, and temp files all get copied into the image. Always exclude dev artifacts.

__pycache__/ *.pyc .env .git .gitignore venv/ .venv/ .DS_Store *.egg-info/
Impact: Without .dockerignore, a 5MB Flask app balloons to 200MB+ in the build context.

Common Mistakes + Fixes

Real errors from real Cloud Run deployments — symptom, root cause, fix.
1
Missing gunicorn

Symptom: Container crashes under load. Root cause: CMD uses python main.py. Fix: Use gunicorn CMD.

2
PORT mismatch

Symptom: Health checks fail, container restarts. Root cause: App listens on 5000, Cloud Run expects 8080. Fix: os.environ.get("PORT", 8080).

3
No .dockerignore

Symptom: 2+ min builds, 500MB images. Root cause: venv, __pycache__, .git copied. Fix: Add .dockerignore.

4
127.0.0.1 vs 0.0.0.0

Symptom: Works locally, fails on Cloud Run. Root cause: host="127.0.0.1" blocks external connections. Fix: host="0.0.0.0".

5
Wrong CMD format

Symptom: Container exits immediately. Root cause: Shell form doesn't forward signals. Fix: Exec form CMD ["gunicorn", ...].

6
Layer order wrong

Symptom: Code change triggers full pip reinstall. Root cause: COPY . . before RUN pip. Fix: COPY requirements.txt first.

3-Minute Technical Walkthrough
Architecture: Internet → Cloud Run HTTPS proxy → container → gunicorn master/worker → Flask router → handler → JSON. Cloud Run scales from 0 to N.

Key Decisions:
  • Slim base image: python:3.11-slim (120MB vs 1GB+), faster cold starts (2-3s vs 10s+).
  • Gunicorn over Flask dev server: Dev server is single-threaded. Gunicorn forks workers for concurrent requests.
  • Layer caching: COPY requirements.txt first installs deps only when requirements changes. Code-only builds in ~5s.
  • Health check: Cloud Run polls /health. Non-200 = instance killed and restarted. Auto-healing without extra config.
