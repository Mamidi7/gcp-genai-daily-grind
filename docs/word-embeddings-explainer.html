Why Words Are Points in Space

How word embeddings turn language into numbers, and why "king minus man plus woman equals queen" is not magic — it is geometry.

The One-Sentence Story

Core Idea

Words become vectors

Every word is turned into a list of numbers (a vector). Words with similar meanings end up close together in that number-space.

Problem

Computers cannot read

A computer does not know that "king" and "monarch" are related. Raw text is just bytes. We need a numerical representation.

Solution

Train embeddings from context

A neural network learns to predict a word from its neighbours. The hidden layer weights become the embedding vector.

How It Works

Think of a thesaurus. "King", "queen", "prince", "monarch" belong on the same page. "Apple", "banana", "pear" belong on another.

Embeddings do the same thing, but in 300-dimensional space instead of paper pages. You cannot draw 300 dimensions, so below we show a 2D projection — the same idea, flattened for your screen.

Key insight: The distance between two embedding vectors measures semantic similarity. Close = similar meaning. Far = different meaning.

The Classic Example

This is the most famous embedding arithmetic. It works because the model learned the gender direction and the royalty direction as separate axes in vector space.

# Word embedding arithmetic (conceptual)
import numpy as np

# Each vector is 300 numbers long
king_vec   = embeddings["king"]      # shape: (300,)
man_vec    = embeddings["man"]
woman_vec  = embeddings["woman"]

# Vector arithmetic
result_vec = king_vec - man_vec + woman_vec

# Find the closest word to this new vector
closest_word = nearest_neighbor(result_vec)
# Output: "queen"

How Are They Trained?

The simplest mental model: predict the missing word.

Take a sentence. Mask one word. Train a neural network to guess it from the words around it. After training millions of sentences, the network's internal weights are the embeddings.

Two famous approaches:

Word2Vec (Mikolov, 2013) — predicts a word from its neighbours. Fast, simple, still useful.

BERT (Devlin, 2018) — predicts masked words using deep bidirectional context. State-of-the-art for many tasks.

# GCP Vertex AI — get text embeddings in production

model = TextEmbeddingModel.from_pretrained("text-embedding-004")

# Batch embed — real production pattern
inputs = ["king", "queen", "man", "woman"]
embeddings = model.get_embeddings(inputs)

for text, emb in zip(inputs, embeddings):
    print(f"{text}: {len(emb.values)} dimensions")
# king: 768 dimensions

Why This Matters for RAG

Retrieval-Augmented Generation (RAG) needs to find the right document chunks to feed an LLM. It does this by:

1. Embedding the user's question into a vector.
2. Searching a vector database for the closest document chunks.
3. Passing those chunks as context to the LLM.

If the embeddings are poor, the wrong chunks come back. The LLM hallucinates. The whole pipeline fails.

# BigQuery VECTOR_SEARCH — zero-infrastructure RAG retrieval
SELECT
  query.id AS question_id,
  base.text AS matched_chunk,
  distance
FROM VECTOR_SEARCH(
  TABLE `project.dataset.doc_embeddings`,   -- base table
  'embedding',                                    -- embedding column
  (SELECT embedding FROM `project.dataset.query_embeddings` WHERE id = 1),
  top_k => 5
) AS result

Interview Corner

30-Second Pitch

"Word embeddings turn text into dense vectors. Similar words cluster together. You can do arithmetic: king minus man plus woman gives you a vector closest to queen. In production, I use Vertex AI's text-embedding-004 and BigQuery VECTOR_SEARCH for retrieval."

90-Second STAR

Situation: Building a RAG pipeline for technical docs.
Task: Needed retrieval that understood semantic similarity, not just keyword match.
Action: Chose text-embedding-004, chunked docs at 512 tokens, stored in BigQuery with VECTOR_SEARCH.
Result: Retrieved relevant chunks even when user queries used different words.

Deep Follow-Up

"How would you handle a query that uses slang or domain jargon the embedding model never saw?"

Answer: Fine-tune on domain corpus, or use hybrid search (BM25 + vector), or add query expansion with an LLM rewrite step.
