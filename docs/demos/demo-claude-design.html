Architecture

Browser → Cloud Run → Gunicorn → Flask — four layers, one request path.

Layer 1

Browser / Client

Sends HTTP request. Could be curl, web app, mobile client, or another service calling your API.

Layer 2

Cloud Run

Managed compute. Receives request, starts container if cold, routes to Gunicorn on port 8080.

Layer 3

Gunicorn

Production WSGI server. Manages worker processes, handles concurrent requests, forwards to Flask.

Layer 4

Flask App

Your Python application. Route handlers, business logic, response generation.

Code

The files that make it work — and why each one matters.

main.py
# Flask app with health endpoint from flask import Flask app = Flask(__name__) @app.route("/") def hello(): return {"message": "Hello from Flask on Cloud Run!"} @app.route("/health") def health(): return {"status": "healthy"} if __name__ == "__main__": app.run(host="0.0.0.0", port=8080)
Why Gunicorn?

Flask's dev server is single-threaded. Under concurrent load on Cloud Run, it crashes. Gunicorn manages worker processes and handles concurrent requests properly.

CMD Format Matters

Use exec form (CMD ["gunicorn", "--bind", "0.0.0.0:8080", "main:app"]) not shell form. Shell form doesn't handle SIGTERM cleanly during scale-down.
