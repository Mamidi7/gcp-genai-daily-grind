The Question
How to re-run only Task 5 (and its downstreams) without re-executing T1-T4 which already succeeded?

Naive approach: Restart the entire pipeline. T1-T4 run again — waste of time, compute, and money. Smart approach: Use one of the 4 strategies below to resume from the failure point.

02 Strategy A — DAG-Based Pipeline Rerun

Strategy A
Cache + Selective Re-execution
The pipeline tracks every task's output as an artifact. When a task fails, you fix the issue and re-run only the failed task + downstream tasks. Tasks that already succeeded (T1-T4) reuse cached results. This is how Vertex AI Pipelines works.
T1→ T2→ T3→ T4→ T5 (rerun)→ T6 (rerun)→ T7 (rerun)→ T8 (rerun)→ T9 (rerun)→ T10 (rerun)

Green = cached (input unchanged) · Red = failed, now fixed · Gold = downstream, needs fresh run

✔ Minimally invasive — only re-runs what's needed
✔ Vertex AI Pipelines has built-in "rerun task" feature (preview)
✔ Maintains original run reference for traceability
✖ Requires pipeline framework (Kubeflow, Vertex Pipelines)
✖ Caching logic can get complex with non-deterministic steps
✖ Still in preview on Vertex AI
# Vertex AI Pipelines — rerun a specific task from google.cloud import aiplatform # Get the failed pipeline run pipeline_job = aiplatform.PipelineJob.get(resource_name="projects/.../pipelineJobs/...") # Rerun from task 5 — Vertex reuses cache for T1-T4 # Only T5 (failed) + downstream tasks re-execute pipeline_job.rerun(task_name="task-5") # Modifies config, re-runs downstream

03 Strategy B — Checkpoint & Resume

Strategy B
Periodic State Persistence
Each task writes partial progress to persistent storage (GCS, BigQuery, Cloud SQL). When a task fails, on restart it checks for existing checkpoint data and resumes from where it left off instead of starting from scratch. This is the recommended pattern for Cloud Run Jobs.

How it works:

Task starts, checks GCS for a checkpoint file: gs://my-bucket/pipeline/task5-checkpoint.json
If checkpoint exists, restore state and continue from last committed offset
Every N records / N seconds, update the checkpoint with current progress
On successful completion, delete checkpoint or write a "done" marker
✔ Works with any task runner — Cloud Run Jobs, Batch, Compute Engine
✔ No special pipeline framework needed
✔ Fault-tolerant by design — even VM restart doesn't lose progress
✖ Requires application-level checkpointing logic
✖ Extra complexity in writing and reading checkpoint state
✖ Not suitable for non-idempotent or stateful operations
import json from google.cloud import storage client = storage.Client() bucket = client.bucket("my-pipeline-checkpoints") checkpoint = bucket.blob("pipeline/task5-checkpoint.json") # On start — try to load checkpoint last_offset = 0 if checkpoint.exists(): state = json.loads(checkpoint.download_as_text()) last_offset = state["offset"] print(f"Resuming from offset {last_offset}") # Process records from last_offset onwards for i in range(last_offset, total_records): process_record(records[i]) # Every 100 records, save checkpoint if i % 100 == 0: checkpoint.upload_from_string( json.dumps({"offset": i, "status": "in_progress"}) ) # On success, mark complete checkpoint.upload_from_string(json.dumps({"status": "completed"}))

04 Strategy C — Idempotent Task Retry

Strategy C
Safe Re-execution via Idempotency
Design every task so that running it multiple times produces the same result. When T5 fails, just retry it (manually or via lifecycle policy). Since each task is idempotent, T1-T4 are untouched — re-running them would produce the same output anyway, so you skip them and re-run only T5. This is the recommended approach for Google Cloud Batch.

Lifecycle Policies (Batch):

# Batch job with lifecycle policy — auto-retry task on failure { "taskGroups": [{ "taskSpec": { "runnables": [{"script": {"text": "python task5.py"}}], "maxRetryCount": 3, "lifecyclePolicies": [{ "action": "RETRY_TASK", "actionCondition": { "exitCodes": [50001, 50002] # Spot VM preemption, network errors } }] } }] }

Idempotency patterns:

Use unique keys (dedup by ID) Write output to new location, not in-place Check IF NOT EXISTS before creating Use INSERT ON CONFLICT / MERGE in SQL
✔ Simplest to implement — just retry the failed task
✔ Batch lifecycle policies do this automatically
✔ Works for transient failures (network, preemption, quota)
✖ Only for transient failures, not logic bugs
✖ Requires all tasks to be truly idempotent
✖ Doesn't help if T5's output is needed by downstream but T5 is broken

05 Strategy D — Saga Pattern

Strategy D
Compensating Transactions + Retry
For distributed transactions spanning multiple services (e.g., payment + inventory + shipping), the Saga pattern pairs each task with a compensating action that undoes it. When T5 fails, the saga executes compensating actions for T5's side effects (rollback), then retries T5 with corrected input. GCP Workflows supports this natively.
T1→ T2→ T3→ T4→ T5 (fail)
↴ Execute compensating action for T5 (undo partial work)
T5 (retry)→ T6→ T10

GCP Workflows — try/retry block:

# Workflows YAML — retry with exponential backoff main: steps: - task5: try: call: http.post args: url: https://my-service/task5 body: {data: "${input}"} result: task5_result retry: predicate: ${custom_retry} max_retries: 3 backoff_duration: 5s backoff_factor: 2.0 except: as: e steps: - compensate_task5: call: http.post args: url: https://my-service/task5/compensate - raise: ${e}
✔ Handles distributed transactions with side effects
✔ Workflows supports native retry + saga pattern
✔ Compensating actions ensure data consistency
✖ Most complex to implement — need compensating logic for every step
✖ Compensating actions may also fail (requires idempotent compensation)
✖ Overkill for simple sequential batch processing

06 When to Use Which — Comparison

Strategy Best For GCP Service Complexity Recovery Scope
A: DAG Rerun ML pipelines, data pipelines with clear DAG structure Vertex AI Pipelines Medium Task 5 + downstream
B: Checkpoint + Resume Long-running batch jobs, data processing with large volumes Cloud Run Jobs, Batch Medium-High Within Task 5 (offset-level)
C: Idempotent Retry Transient failures (network, VM preemption, quota) Batch lifecycle policies Low Task 5 only (full retry)
D: Saga Pattern Distributed transactions, multi-service orchestration Workflows High Task 5 + compensate + retry
E: Airflow Clear DAG-based pipelines, multi-cloud orchestration, cloud resource provisioning Cloud Composer / Managed Airflow Low-Medium Task 5 + downstream (Clear with Downstream) or Task 5 only (Clear Current)
Recommendation
Start with Strategy C (idempotent retry) for transient failures. Add Strategy B (checkpointing) when tasks process large datasets. For complex ML pipelines, use Strategy A (Vertex AI Pipelines rerun). For multi-service transactions, use Strategy D (Workflows saga).

Multi-cloud by design 300+ operators Not just data pipelines Managed services on all clouds
The "Clear" Mechanism
How Airflow Handles Failed Task Reruns
In Airflow, when a task fails in a DAG, you don't "rerun" it — you Clear it. Clearing a task instance resets its state to "None", which tells the scheduler to re-execute it on the next DAG run. Airflow gives you two options when clearing:
Option 1
Clear with Downstream
Clears the failed task plus all tasks downstream of it. Upstream tasks (already succeeded) are not cleared — their outputs are preserved.

Best for: When the failure may affect downstream logic, or when downstream tasks also need fresh data from the corrected task.
Option 2
Clear Current Task Only
Clears only the failed task. Upstream and downstream tasks are untouched. If downstream tasks already succeeded with stale data, they stay as-is.

Best for: Transient failures where only the failed task needs retrying. Faster recovery since downstream doesn't re-run.
Key insight: In Airflow, "Clear" = Rerun. The word "rerun" doesn't exist in the Airflow UI — you clear the task instance's state, and the scheduler picks it up naturally. This is identical to the question "how to rerun task 5 without affecting upstream tasks" — Clear with Downstream is exactly the answer: it preserves T1-T4 (upstream), re-executes T5 (failed), and re-executes T6-T10 (downstream).

09 Interview Prep

"To rerun only the 5th task without affecting upstream tasks, use a DAG-based pipeline system with output caching (like Vertex AI Pipelines) where tasks 1-4 reuse cached outputs, and only task 5 + downstream re-execute. Alternatively, design tasks to be idempotent so you can safely retry task 5 alone. For long-running tasks, use checkpointing where partial progress is saved to persistent storage so the task resumes from its last committed offset rather than from scratch."

Situation: Production data pipeline with 10 sequential tasks processing customer transactions. Task 5 (data enrichment) failed at 2 AM due to a dependency timeout.

Task: Resume the pipeline from task 5 without re-processing tasks 1-4 (which took 3 hours).

Action: Used Vertex AI Pipelines rerun — T1-T4 outputs were cached. Fixed the timeout issue in task 5's config (increased timeout from 30s to 120s). Triggered rerun from task 5. Pipeline skipped T1-T4 (cache hit) and re-ran T5 through T10.

Result: Pipeline completed in 45 minutes instead of 4 hours. Recovery required zero data reprocessing. Added lifecycle policies for automatic retry on timeout errors for future runs.

Depth 1 — Problem analysis: The 10 tasks form a sequential DAG. Task 5 failed. Tasks 1-4 produced deterministic outputs (idempotent + no side effects). The naive restart wastes 3 hours of compute. The solution must: (a) preserve T1-T4 results, (b) re-run T5 with fixes, (c) re-run T6-T10 with T5's fresh output.

Depth 2 — Strategy comparison: Strategy C (idempotent retry) works if the failure is transient — just retry T5. Strategy B (checkpointing) works if T5 processes data in batches — resume from last checkpoint. Strategy A (DAG rerun) works when the pipeline framework supports caching — reuse upstream artifacts. Strategy D (saga) is for distributed transactions with side effects.

Depth 3 — GCP implementation details: Vertex AI Pipelines rerun (preview) — call pipeline_job.rerun(task_name="task-5"). Cloud Run Jobs — use checkpointing to GCS with BATCH_TASK_RETRY_ATTEMPT env var. Batch — lifecycle policies with maxRetryCount: 3 and exit code filters. Workflows — try/retry with backoff_duration and compensating steps.

Depth 4 — Failure modes: Non-deterministic tasks break caching (always use fixed seeds). Checkpoint writes may fail (write to multiple locations). Compensating actions may also fail (use idempotent compensation with retries). Cache invalidation is tricky (version input schemas).

Depth 5 — Design principles: Always design pipelines with idempotency as a first-class concern. Output immutability (write-once) makes caching natural. Deterministic tasks enable safe retries. Explicit failure modes (via exit codes) allow lifecycle policies to handle different failures differently.

Possible Interview Questions

Q: Your pipeline task failed after 2 hours of processing. How do you recover?
A: First, assess if the failure is transient (network, preemption) or permanent (code bug, data issue). For transient, use lifecycle policies with automatic retry (Batch) or checkpointing (Cloud Run Jobs). For permanent, use DAG rerun (Vertex AI Pipelines) with output caching to avoid redoing upstream work. Always design for idempotency so retries are safe.
Q: What's the difference between retry and rerun?
A: Retry re-executes the same task with the same inputs, expecting transient issues to resolve. Rerun implies you may modify config or fix bugs before re-executing. Retry is automatic (lifecycle policies), rerun is typically manual or triggered after a code/config change.
Q: How do you handle side effects in retryable tasks?
A: Use the Saga pattern — each task has a compensating action. If T5 sends an email and then fails, the compensation undoes the email (or marks it as rolled back). Pair this with idempotency keys so retries don't send duplicate emails. GCP Workflows has native support for this pattern.
Q: In Airflow, how do you rerun a failed task without affecting upstream tasks?
A: Use Clear with Downstream. In Airflow, "Clear" resets a task instance's state to None, so the scheduler re-executes it. Clear with Downstream resets the failed task AND all downstream tasks — upstream tasks stay untouched. This is exactly like our 10-task scenario: T1-T4 are preserved, T5 re-runs, and T6-T10 also re-run with T5's fresh output. To rerun only T5 alone, use Clear Current Task Only.
Q: Is Airflow only for data pipelines? Can you use it for multi-cloud resource provisioning?
A: No, Airflow is not limited to data pipelines. It's a general-purpose workflow orchestrator with 300+ operators for GCP, AWS, Azure, and others. You can use it to provision VMs, deploy Kubernetes clusters, manage IAM, run Terraform, trigger Cloud Run deployments — anything that has an API. Each cloud also has a managed Airflow service: Cloud Composer (GCP), MWAA (AWS), and Managed Airflow (Azure).
