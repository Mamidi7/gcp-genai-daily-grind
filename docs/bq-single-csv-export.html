The Problem

Why your export takes >1 hour and disconnects kill it
Root Cause 1
Single CSV limit: 1GB compressed / ~10-15GB uncompressed
Root Cause 2
Distributed system → single file bottleneck — all workers queue up
Root Cause 3
Python client polls job.status() — disconnect = no polling = job.result() raises
Root Cause 4
Partitioned tables — each partition is separate storage, export scans ALL
BigQuery Export Limitation
BQ exports to a single CSV file are limited to 1GB (compressed). Beyond that, the job silently fails or runs indefinitely. The distributed system was designed for parallel output — forcing single-file output is like asking 100 chefs to cook into one bowl.
The Disconnect Problem

When your workstation disconnects midway through job.result():

  • The Python client was polling BQ every few seconds for job status
  • Disconnect = socket closes = TCP connection to BQ API drops
  • job.result() throws ConnectionError or TimeoutError
  • The export job on BQ continues running (server-side), but your client lost track
  • You don't know if it finished, failed, or is still running

SOLUTION 1 • RECOMMENDED
🎯 Wildcard Export + gsutil compose
The single most impactful change. Instead of exporting to one file, tell BQ to write multiple files using a wildcard pattern. BQ splits the data across parallel workers, each writing its own 1GB chunk. Then merge them locally or in GCS.
✅ 5-20x speedup ✅ No timeout ⚠️ Need compose step ✅ Disconnect safe

Python Code

# === STEP 1: Export with wildcard === from google.cloud import bigquery client = bigquery.Client() table_ref = "my-project.my_dataset.my_table" # KEY CHANGE: Use wildcard in the filename # BQ auto-splits into ~1GB files: output-000000000000.csv, etc. destination = "gs://my-bucket/exports/output-*.csv" extract_job = client.extract_table( table_ref, destination, job_config=bigquery.ExtractJobConfig( destination_format=bigquery.DestinationFormat.CSV, print_header=True ) ) extract_job.result() # Returns in minutes, not hours! print(f"Exported {extract_job.output_rows} rows")
BQ splits the data across distributed workers — each worker writes its own chunk. A 50GB table generates ~10-15 files, each complete with headers.

Step 2: Merge with gsutil compose (if you NEED one file)

# List the generated files gsutil ls gs://my-bucket/exports/output-*.csv # Create a manifest file and compose # Note: gsutil compose max 32 components at a time gsutil compose \ gs://my-bucket/exports/output-000000000000.csv \ gs://my-bucket/exports/output-000000000001.csv \ ... gs://my-bucket/exports/output-merged.csv # Or use a script to auto-generate the command python3 -c """ import subprocess files = subprocess.check_output( ['gsutil','ls','gs://my-bucket/exports/output-*.csv'] ).decode().strip().split('\n') parts = ' '.join(files) subprocess.run(f'gsutil compose {parts} gs://my-bucket/exports/merged.csv', shell=True) print('Composed successfully') """
Pitfall: gsutil compose has a 32-component limit
If you have >32 files, compose in batches: merge 32 at a time into intermediate files, then merge those intermediates. Or download locally and use cat (Unix) to concatenate CSV files (skip headers on all but first).
SOLUTION 2
📋 EXPORT DATA Statement (SQL-based)
If you prefer writing SQL over Python API calls, use the EXPORT DATA DDL statement. This runs server-side and handles partitioning elegantly with a WHERE _PARTITIONTIME filter.
✅ Self-contained SQL ✅ Partition filter built-in ⚠️ Requires wildcard URI ✅ Disconnect safe

Basic Export (Full Table)

-- Export entire table to wildcard CSV files EXPORT DATA OPTIONS ( uri = 'gs://my-bucket/exports/data-*.csv', format = 'CSV', overwrite = TRUE, header = TRUE ) AS SELECT * FROM `my-project.my_dataset.my_table`;

Partition-Specific Export (One Day)

-- Export only one day's partition EXPORT DATA OPTIONS ( uri = 'gs://my-bucket/exports/20240115-*.csv', format = 'CSV', overwrite = TRUE, header = TRUE ) AS SELECT * FROM `my-project.my_dataset.my_table` WHERE _PARTITIONTIME = TIMESTAMP("2024-01-15");
The EXPORT DATA statement runs entirely server-side. Your client can disconnect — the job continues on BQ. To check status, query INFORMATION_SCHEMA.JOBS_BY_USER later.
SOLUTION 3
🔄 Resumable Client — Save Job ID, Resume Anywhere
If you truly need a direct download (no GCS intermediate) or can't use compose, make your export resilient to disconnection. The trick: assign your own job ID, save it, and rebuild the client from another machine.
⚠️ No speedup ✅ Survives disconnect ⚠️ Single CSV limit still applies

Export with Custom Job ID

from google.cloud import bigquery import json client = bigquery.Client() table_ref = "my-project.my_dataset.my_table" destination = "gs://my-bucket/exports/output.csv" # Assign a custom job ID so we can reference it later job_id = "export_20240115_v1" job_config = bigquery.ExtractJobConfig() job = client.extract_table( table_ref, destination, job_id=job_id, job_config=job_config ) # Save job info BEFORE waiting with open("bq_export_checkpoint.json", "w") as f: json.dump({"project": client.project, "job_id": job_id, "location": "US"}, f) # Try to wait — if disconnect happens, run resume script try: job.result(timeout=600) # 10 min timeout print("Export completed") except Exception as e: print(f"Connection lost: {e}") print("Run resume script on another machine")

Resume Script (Run from another machine)

# resume_export.py — Run on cloud shell or another VM import json from google.cloud import bigquery with open("bq_export_checkpoint.json") as f: checkpoint = json.load(f) client = bigquery.Client(project=checkpoint["project"]) # Fetch the existing job by ID job = client.get_job( job_id=checkpoint["job_id"], location=checkpoint["location"] ) if job.state == "DONE": print("Job already finished!") print(f"Destination: {job.destination_uris}") elif job.state == "RUNNING": print("Job still running, waiting...") job.result() # Resume polling print("Done!") else: print(f"Job state: {job.state}")
Key Insight: BQ Jobs have a 7-day lifetime
Even if your client disconnects, the BQ job continues running server-side for up to 7 days (or until completion). You can always retrieve it with client.get_job(job_id) from any machine with valid credentials.
SOLUTION 4
📅 Partition-by-Partition Export
For partitioned tables (your use case), export one partition at a time. Each partition is small enough to finish in minutes. If one fails, you only redo that day. Concatenate all files at the end.
✅ Built-in checkpoint ✅ Very fast per partition ⚠️ Many files to merge ✅ Disconnect = resume from last partition

Python Code

from google.cloud import bigquery from datetime import date, timedelta import json, os client = bigquery.Client() project = "my-project" dataset = "my_dataset" table = "my_table" # Assume today's date as reference start_date = date(2024, 1, 1) end_date = date(2024, 1, 31) checkpoint_file = "export_checkpoint.json" # Load checkpoint to resume from last failure completed = set() if os.path.exists(checkpoint_file): with open(checkpoint_file) as f: completed = set(json.load(f).get("completed", [])) current = start_date while current <= end_date: day_str = current.strftime("%Y%m%d") if day_str in completed: current += timedelta(days=1) continue # Export one partition: table_name$YYYYMMDD table_ref = f"{project}.{dataset}.{table}${day_str}" dest = f"gs://my-bucket/exports/{day_str}.csv" print(f"Exporting {day_str}...") job = client.extract_table(table_ref, dest) job.result() # Each partition is fast — 2-5 min print(f" Done: {job.output_rows} rows") # Save checkpoint after each partition completed.add(day_str) with open(checkpoint_file, "w") as f: json.dump({"completed": sorted(completed)}, f) current += timedelta(days=1) print("All partitions exported!")
Each partition is a separate small job. If disconnect happens at partition 15/31, run again and it starts from partition 16. The checkpoint file tracks progress.

Task: The Python API export was taking >1 hour, and workstation disconnects killed the job midway, requiring a full restart.

Action: Changed the export destination from output.csv to output-*.csv, letting BQ split the work across its distributed workers. Added a second step using gsutil compose to merge the parts. 3-Minute Technical Walkthrough

Architecture: BigQuery's export API has an important design constraint — single CSV destinations can receive at most 1GB of compressed data because the export must route through a single writer node. For distributed systems, this creates a bottleneck where all workers serialize their output to one stream.

Parallel approach: By using a wildcard URI (e.g. gs://bucket/data-*.csv), each BQ worker writes its own shard in parallel. BQ automatically determines the number of shards based on data size. This reduced our 50GB export from 75 min to 6 min.

Merging shards: gsutil compose concatenates GCS objects without downloading. But there's a 32-component limit — for larger exports, I built a recursive compose script that merges in batches of 32, then merges the intermediates.

Resilience layer: The custom job ID pattern lets us detach the client from the job. Even if the laptop goes to sleep, the BQ job keeps running. A Cloud Shell script can pick it up with client.get_job(). The job state persists for 7 days.

Partition strategy: For the daily partitioned case specifically, iterating partition-by-partition adds built-in checkpointing — each day is a separate job, so a failure on day 20 only restarts from day 20, not day 1.
