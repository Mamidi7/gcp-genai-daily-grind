SQL Deep Dive

Two steps: create the index, then search by meaning.
▼

Step 1: Create Vector Index

-- Create a vector index on the embedding column CREATE VECTOR INDEX my_index ON my_dataset.my_table(embedding) OPTIONS(distance_type='COSINE', index_type='IVF', ivf_options='{"num_lists":100}');
What this does
Creates an approximate nearest neighbor (ANN) index on the embedding column. Without this, VECTOR_SEARCH does a brute-force scan — fine for small tables, slow for millions of rows. IVF = Inverted File Index, divides vectors into 100 clusters for faster search.

Step 2: Search with VECTOR_SEARCH()

SELECT base.text, base.category, query.query_text, distance FROM VECTOR_SEARCH( TABLE my_dataset.my_table, 'embedding', ( SELECT embedding AS query_embedding, 'fraud detection policy' AS query_text FROM ML.GENERATE_EMBEDDING( MODEL my_dataset.embedding_model, (SELECT 'fraud detection policy' AS content) ) ), top_k => 5 );
Key parameters
TABLE = the table with your embeddings. Second arg = column name as string. Subquery = generates the query embedding. top_k = how many results to return. distance_type must match the index (COSINE in this case).

More Efficient: Pre-Embed the Query

-- For production: embed query separately, pass directly SELECT base.text, base.category, distance FROM VECTOR_SEARCH( TABLE my_dataset.my_table, 'embedding', (SELECT [0.012, -0.045, 0.078, /* ... 768 floats ... */] AS query_embedding), top_k => 5, distance_type => 'COSINE' );
⚠️ Gotcha — Query Embedding

If you use ML.GENERATE_EMBEDDING inside the VECTOR_SEARCH call, it re-embeds the query for every row scan. Better to pre-embed the query in your app code and pass the vector directly. This also lets you reuse cached embeddings.

Traditional vs Vector Search

Same query, different approaches — see the gap.
▼

❌ LIKE / Text Search

  • WHERE text LIKE '%fraud%'
  • Misses "scam", "unauthorized transaction"
  • Case-sensitive (unless LOWER)
  • No ranking by relevance
  • Can't handle typos: "fraaud" → no match
  • Exact word match only

✅ VECTOR_SEARCH

  • VECTOR_SEARCH(TABLE t, ...)
  • Finds "scam", "fraud", "unauthorized tx"
  • Semantic understanding
  • Returns ranked by distance (relevance)
  • Handles typos — meaning is preserved
  • Language agnostic (telugu query kuda work chestundi)

Interview Prep

Three levels of answers — pick your depth.
▼
🎯 30-Second Answer

"BigQuery Vector Search lets you find text by meaning, not keywords. You store embeddings (from text-embedding-004) as a column in BigQuery, create a vector index, and use the VECTOR_SEARCH() SQL function. No separate vector database needed — everything stays in BigQuery."

🎯 90-Second STAR Answer

Problem: We needed semantic search on a corpus of 500K banking transaction descriptions. Traditional LIKE queries missed synonyms and typos — "fraud" wouldn't match "unauthorized tx."

Action: I implemented BigQuery Vector Search. I embedded all descriptions using text-embedding-004 (768-dim vectors), stored them as FLOAT64[] in a new column, created a COSINE IVF index, and wrote VECTOR_SEARCH() queries with top_k=10. The embedding happens in a weekly batch job via a Cloud Function.

Result: 40% improvement in relevant result recall. Zero additional infrastructure — no Pinecone, no Chroma. Query latency under 2 seconds for 500K rows. Production ready with zero ops overhead.

🎯 3-Minute Technical Walkthrough

Architecture: BigQuery table with columns (id, text, category, embedding FLOAT64[]). Embedding generated via Vertex AI text-embedding-004, stored as repeated float array. Vector index created with IVF (Inverted File Index) using cosine distance. Query-time: embed user query → VECTOR_SEARCH() → ranked results.

Key decisions: (1) IVF not Brute Force — IVF is approximate but 10x faster for large data. (2) COSINE not EUCLIDEAN — cosine works better for text embeddings. (3) Pre-embedding — embed queries in app code, not inline in SQL.

Tradeoffs: BigQuery Vector Search is slower than dedicated vector DBs (Pinecone) for real-time use cases. But for batch/analytical workloads where data already lives in BQ, it eliminates ETL overhead. Best for: RAG on historical data, document search, classification. Not for: real-time recommendation (sub-100ms needed).

What can fail: Index not yet built, embedding model mismatch, column type mismatch, distance_type mismatch. Always verify index status via INFORMATION_SCHEMA.
