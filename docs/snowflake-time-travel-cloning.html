Time Travel & Zero-Copy Cloning

The two Snowflake features that no other warehouse has — and every interviewer asks about.

2. Time Travel — 90 Days Back

Any point. Query. Clone. Undrop. Without backup restore.

How It Works

Snowflake uses immutable micro-partitions — 16MB compressed columnar files. UPDATE or DELETE chestene? Snowflake doesn't touch existing files. It creates new partitions and marks old ones inactive.

Before UPDATE: ████ ████ ████ ████ (4 active) │ UPDATE salary → │ UPDATE never mutates — it appends ▼ After: ████ ████ ████ ████ ██████ └───── active ─────┘ └─ new ┘ ░░░░ ░░░░ ░░░░ (old = Time Travel)

Time Travel period = how long before Snowflake garbage-collects inactive partitions. Fail-safe = extra grace period for Snowflake's internal recovery.

Micro-partition internals — why Time Travel costs storage
Each micro-partition stores ~16MB of uncompressed data per column. Metadata (min, max, null count) stored per partition for pruning. When you UPDATE, Snowflake creates new micro-partitions only for changed columns. Unchanged columns remain shared. So Time Travel storage cost depends on how much data you modified, not total table size.

3. Zero-Copy Cloning — Instant, No-Cost

10TB table ni 0 bytes storage lo clone. Seconds lo ready.

┌──────────────┐ │ Original │── data blocks ──┐ │ (10GB) │ │ └──────────────┘ │ Shared │ │ Immutable │ CLONE (instant, 0 bytes) │ Blocks ▼ │ ┌──────────────┐ │ │ Clone │── writes → │ New blocks for │ (0GB start) │ new blocks │ changed data only └──────────────┘ │ ▼ ════════════════════════════════════════════════ ▓▓ Shared micro-partitions (unchanged) ▓▓ ░░ New partitions (clone's writes only) ░░
clone-patterns.sql
-- Clone a table (zero storage cost)
CREATE TABLE prod_backup CLONE prod_table;

-- Clone a schema
CREATE SCHEMA dev_schema CLONE prod_schema;

-- Clone as-of specific time (most powerful pattern)
CREATE TABLE before_deploy CLONE orders
  AT (TIMESTAMP => '2026-06-01 08:00:00');

Use Cases

  • Dev/Test — clone prod, zero storage, real data
  • Backup before risky ops — clone before DDL, rollback instantly
  • CI/CD pipeline — CREATE DATABASE dev CLONE prod; → migrate → validate → drop
  • Point-in-time analysis — clone + AT together
Storage Detail: Clone = 0 bytes initially. Billed only when data diverges (new micro-partitions for changed data). Shared partitions remain shared until changed.

4. SQL Syntax — Three Ways to Travel

AT + TIMESTAMP / OFFSET / STATEMENT. Each for a different scenario.

01-travel-syntax.sql
-- 1. Exact point in time
SELECT * FROM orders
  AT (TIMESTAMP => '2026-06-01 10:00:00');

-- 2. Relative offset (negative seconds from now)
SELECT * FROM orders
  AT (OFFSET => -3600);  -- 1 hour ago

-- 3. By statement ID (from Query History)
SELECT * FROM orders
  AT (STATEMENT => '0190a5a5-0000-...');
02-recovery-patterns.sql
-- Recover a dropped table (easiest)
UNDROP TABLE orders;

-- Restore table as it was 1 day ago
CREATE TABLE orders_restored AS
  SELECT * FROM orders
  AT (OFFSET => -86400);

-- Row-level restore: merge specific rows
MERGE INTO orders t
USING (SELECT * FROM orders AT (OFFSET => -3600)) s
  ON t.id = s.id
WHEN MATCHED AND t.status = 'bad'
  THEN UPDATE SET t.status = s.status;
AT vs BEFORE — Confusion Point
AT (STATEMENT => 'id') = data at the time the statement completed (includes its changes).
BEFORE (STATEMENT => 'id') = data immediately before the statement ran (excludes its changes).
Use BEFORE when you want to undo a specific operation.

5. Common Mistakes + Fixes

Interview traps and production gotchas.

mistake-1-dropped-table.sql
-- ❌ WRONG: Direct clone on dropped table
CREATE TABLE restored CLONE my_table;
-- Error: Object 'my_table' does not exist.

-- ✅ RIGHT: Clone from time BEFORE drop
CREATE TABLE restored CLONE my_table
  AT (TIMESTAMP => '2026-05-15 14:30:00');

-- Or simpler:
UNDROP TABLE my_table;
mistake-2-clone-cost.sql
-- ❌ WRONG: Clone is NOT free forever
CREATE DATABASE prod_copy CLONE prod_db;
-- ...30 days later, storage bill spikes.
-- Clone stays free only until you WRITE to it.

-- ✅ RIGHT: Short-lived clones, drop when done
CREATE DATABASE temp_analytics CLONE prod_db;
-- Run analysis...
DROP DATABASE IF EXISTS temp_analytics;
When NOT to Clone
  • Only need subset of rows? Use CREATE TABLE ... AS SELECT with WHERE
  • PII masking needed? Clone then apply masking policy
  • Long-lived copies? Budget for storage if you modify data

6. Fail-Safe — Last Resort

7 extra days. But you can't query it. Snowflake support only.

Don't rely on Fail-safe. It exists for Snowflake's internal DR, not as a user backup feature. Always use Time Travel + cloning for your recovery strategy.

7. Interview Answers

Three depths — 30s / 90s STAR / 3-min walkthrough.

30-Second Pitch

"Snowflake Time Travel lets me query or restore data as it existed up to 90 days ago using AT/BEFORE syntax. Zero-Copy Cloning creates instant copies without duplicating storage — charges start only when the clone's data changes. Together, they're the fastest recovery path for data accidents."

90-Second STAR

Situation: Production orders table had incorrect status updates from a buggy scheduled task. 6 hours of damage.
Task: Restore 45,000 orders to correct state without losing valid updates in the same window.
Action: Created a zero-copy clone as-of the timestamp before the bug:

CREATE TABLE orders_clean CLONE orders
  AT (TIMESTAMP => '2026-05-20 01:59:59');

Identified affected IDs by diffing current vs snapshot. Used MERGE to update only the status column for affected rows — preserving all other changes from the 6-hour window.
Result: 45,000 orders corrected, zero data loss. Recovery time: ~2 minutes vs 4+ hours traditional restore. Documented as standard recovery playbook.

3-Minute Walkthrough

"Here's how Time Travel works under the hood..."

Snowflake stores data in immutable micro-partitions — ~16MB compressed columnar chunks each. When you UPDATE, Snowflake doesn't touch existing files. It creates new micro-partitions and marks old ones inactive. Those inactive partitions are your Time Travel data.

The retention window (1–90 days) controls how long before Snowflake's background process garbage-collects them.

Zero-copy cloning works the same way — a clone initially points at the exact same micro-partitions as the source. Write to it? Snowflake creates new partitions for only the changed data. The shared 90%+ remains shared.

Common interview question: "You accidentally DROP a table. How do you recover it?"
Answer: UNDROP TABLE tablename; — one command. If Time Travel expired, file a support ticket for fail-safe.

9. Quiz — Test Yourself

5 questions. Click to check your answer.

1. What does AT (BEFORE STATEMENT 'id') return?
Correct: B — BEFORE returns data prior to the statement's changes. AT includes them.
2. Clone a 50GB table. Storage billed immediately?
Correct: C — Zero-copy clone = 0 bytes. Storage billed only on diverged data.
3. Time Travel expired yesterday. Can you still query data from 5 days ago?
Correct: C — Fail-safe exists but requires Snowflake support to recover. Not self-service.
4. Which does NOT consume Time Travel storage?
Correct: C — SELECT is read-only. INSERT/UPDATE/DROP create inactive micro-partitions.
5. Time Travel retention: Standard vs Enterprise?
Correct: A — Standard = 1 day. Enterprise = 90 days. Both have 7 days fail-safe.
