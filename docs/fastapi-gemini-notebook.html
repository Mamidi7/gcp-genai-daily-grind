Architecture — FastAPI meets Gemini

How the app is structured — from HTTP request to Gemini response.
System Overview

A FastAPI server that proxies requests to Vertex AI Gemini

The app has 3 layers: FastAPI routes (receive HTTP requests), Service layer (calls Gemini via google-cloud-aiplatform), and Middleware (logging, CORS, error handling). All I/O is async via asyncio.to_thread().

Deployed on Cloud Run — scales to zero, HTTPS automatically, managed infra.
Figure: Request Lifecycle — Client → Cloud Run → FastAPI → Gemini
⚡
Async First

All Gemini calls wrapped in asyncio.to_thread(). FastAPI handles concurrent requests without blocking.

🔁
Retry Logic

Exponential backoff: 1s → 2s → 4s. Handles Vertex AI quota errors and transient failures gracefully.

✅
Health Probes

/healthz (always 200) + /readyz (checks Gemini) — Cloud Run uses these for rolling updates.

📝
Pydantic V2

Structured input/output with BaseModel. @field_validator over old @validator.

Sync vs Async — How it works

Blocking call: You wait at the counter until the cook finishes your order. If 10 people order, they wait in line. Each waits their full time.

result = vertex_ai_model.generate(prompt) # thread blocks here

In FastAPI, a blocking call blocks the entire event loop — all other requests queue up. That's why we use asyncio.to_thread().
Non-blocking call: You take a number, sit down, and get notified when your order is ready. Meanwhile, 10 other people are also being served.

result = await asyncio.to_thread(vertex_ai_model.generate, prompt)

to_thread() runs the blocking call in a separate thread, freeing the event loop to handle other requests. This is how FastAPI serves many AI calls concurrently.
Figure: Event loop stays responsive while blocking AI calls run in background threads

The Code — Block by Block

Every meaningful block explained: what it does, why it's there, what could fail.
Entry point

main.py — Async routes that wrap Gemini calls

Three endpoints: /healthz (liveness), /readyz (Gemini connectivity check), /chat (proxies user prompt to Gemini). All async. Uses asyncio.to_thread() for the blocking Vertex AI SDK call.
# ===== 1. Imports ===== from fastapi import FastAPI, HTTPException from pydantic import BaseModel, Field import google.cloud.aiplatform as aip # Vertex AI SDK import asyncio, os, logging, json # ===== 2. App ===== app = FastAPI(title="Gemini Chat API") # ===== 3. Pydantic Schema ===== class ChatRequest(BaseModel): prompt: str = Field(..., min_length=1, max_length=4096) temperature: float = Field(default=0.7, ge=0.0, le=2.0) class ChatResponse(BaseModel): reply: str model: str = "gemini-2.0-flash-001" tokens: int | None = None # ===== 4. Async Routes ===== @app.get("/healthz") async def healthz(): return {"status": "alive"} # Always 200 @app.get("/readyz") async def readyz(): try: # Check Gemini connectivity await asyncio.to_thread(test_gemini_connectivity) return {"status": "ready"} except Exception: raise HTTPException(status_code=503, detail="Gemini unavailable") @app.post("/chat") async def chat(req: ChatRequest): try: result = await asyncio.to_thread( generate_with_gemini, req.prompt, req.temperature ) return ChatResponse(reply=result) except google.api_core.exceptions.GoogleAPIError as e: logger.error(json.dumps({"event": "api_error", "detail": str(e)})) raise HTTPException(status_code=503, detail="AI service error") # ===== 5. Entry ===== if __name__ == "__main__": import uvicorn port = int(os.environ.get("PORT", 8080)) uvicorn.run(app, host="0.0.0.0", port=port)
🔑 Key Pattern: await asyncio.to_thread(sync_fn, args) is how you make a blocking library work with async. The Sync vs Async toggle above explains the mental model.
Interview Point
Why asyncio.to_thread() and not just call Gemini directly? Because the Vertex AI SDK is synchronous — it blocks the thread. In FastAPI, blocking the event loop means one slow AI call blocks all other requests. to_thread() offloads it to a thread pool.
Structured I/O

Pydantic V2 — Zero-boilerplate validation

Every endpoint has typed input and output. FastAPI validates automatically. Invalid requests get 422 before reaching your code — fail fast, fail early.
# ===== Request schemas ===== class ChatRequest(BaseModel): prompt: str = Field(..., min_length=1, max_length=4096, description="User prompt for Gemini") temperature: float = Field(default=0.7, ge=0.0, le=2.0) max_tokens: int = Field(default=1024, ge=16, le=8192) @field_validator("prompt") @classmethod def no_empty_prompt(cls, v): if not v.strip(): raise ValueError("Prompt cannot be whitespace-only") return v.strip() class ChatResponse(BaseModel): reply: str model: str = "gemini-2.0-flash-001" tokens: int | None = None latency_ms: float | None = None class ConfigSummary(BaseModel): project: str region: str = "us-central1" model: str endpoints: list[str]
Why @field_validator and not @validator?
Pydantic V2 deprecated @validator in favor of @field_validator. The V2 version is faster, type-safe, and supports @classmethod pattern. In interviews, mention the V2 migration as a real-world upgrade.
Follow-up Question
How does FastAPI know to validate ChatRequest automatically?
When you declare req: ChatRequest in the route handler, FastAPI inspects the type annotation, reads the Pydantic model, and runs validation before your function runs. If validation fails, it returns 422 without executing a single line of your business logic.
Configuration

config.py — Env-driven settings

All config comes from environment variables. No hardcoded secrets. The pydantic-settings pattern validates config at startup.
# ===== config.py ===== from pydantic_settings import BaseSettings class Settings(BaseSettings): project: str # GOOGLE_CLOUD_PROJECT region: str = "us-central1" model: str = "gemini-2.0-flash-001" embed_model: str = "text-embedding-004" location: str = "us-central1" @field_validator("project") @classmethod def project_not_empty(cls, v): if not v: raise ValueError("GOOGLE_CLOUD_PROJECT must be set") return v settings = Settings() # Reads from env
pydantic-settings
Reads from os.environ automatically. Field names map to env vars (project → GOOGLE_CLOUD_PROJECT). If a required field is missing, the app fails at import time, not at request time.
Resilience

Retry Logic — Exponential backoff with jitter

Gemini API calls can fail due to quota limits, transient network issues, or model overload. Rather than letting the request fail, we retry up to 3 times with increasing delays.
# ===== Retry decorator ===== import functools, random, time def retry_with_backoff(max_attempts=3, base_delay=1.0): def decorator(func): @functools.wraps(func) def wrapper(*args, **kwargs): last_error = None for attempt in range(max_attempts): try: return func(*args, **kwargs) except (google.api_core.exceptions. ResourceExhausted, google.api_core.exceptions. ServiceUnavailable) as e: last_error = e if attempt == max_attempts - 1: raise delay = base_delay * (2 ** attempt) + random.uniform(0, 0.5) logger.warning(json.dumps({ "event": "retry", "attempt": attempt + 1, "delay_s": round(delay, 2) })) time.sleep(delay) raise last_error return wrapper return decorator # Usage @retry_with_backoff(max_attempts=3) def generate_with_gemini(prompt: str, temp: float = 0.7): model = GenerativeModel("gemini-2.0-flash-001") response = model.generate_content(prompt) return response.text
🔁 Retry strategy: 1s → 2s → 4s (+ random jitter). Only retries on transient errors (429 ResourceExhausted, 503 ServiceUnavailable). Does NOT retry on auth errors or invalid arguments.
Interview Question
Why add jitter to your backoff?
Without jitter, all retrying clients hit the API at the exact same time (thundering herd). With jitter, they spread out randomly. It's the difference between everyone getting up from their seat simultaneously vs one person at a time.
Cross-cutting

Middleware — Logging, CORS, and Error handling

Middleware wraps every request/response cycle. It adds request IDs for tracing, logs latency, handles CORS, and maps unhandled exceptions to structured error responses.
# ===== Logging middleware ===== import time, uuid @app.middleware("http") async def log_requests(request, call_next): request_id = uuid.uuid4().hex[:8] start = time.time() try: response = await call_next(request) except Exception as e: logger.error(json.dumps({ "event": "unhandled_error", "request_id": request_id, "detail": str(e) })) return JSONResponse( status_code=500, content={"error": "Internal server error", "request_id": request_id} ) latency = time.time() - start logger.info(json.dumps({ "event": "request_complete", "request_id": request_id, "method": request.method, "path": request.url.path, "status": response.status_code, "latency_ms": round(latency * 1000, 1) })) return response # ===== CORS ===== from fastapi.middleware.cors import CORSMiddleware app.add_middleware(CORSMiddleware, allow_origins=["*"], allow_methods=["*"], allow_headers=["*"], )

Common Mistakes + Fixes

Real errors from real development — symptom, root cause, fix.
1
Blocking the Event Loop

Symptom: One slow AI call freezes all other requests. Root cause: Calling Vertex AI SDK directly without to_thread(). Fix: Wrap all sync I/O in asyncio.to_thread().

2
Pydantic V1 Syntax

Symptom: ValidationError with '@validator is deprecated'. Root cause: Using old V1 decorators. Fix: Use @field_validator with @classmethod instead.

3
No Retry on 429/503

Symptom: Random failures under load. Root cause: Gemini API returns ResourceExhausted (429) when quota is hit. Fix: Exponential backoff retry with jitter.

4
Wrong PORT on Cloud Run

Symptom: Health checks fail, container restarts in a loop. Root cause: Hardcoded port vs Cloud Run's env var. Fix: os.environ.get("PORT", 8080) everywhere.

5
Missing GCP Auth

Symptom: Vertex AI returns 403 at runtime. Root cause: No service account or ADC not configured. Fix: Deploy with --service-account flag, or set GOOGLE_APPLICATION_CREDENTIALS.

6
Swallowing Exceptions

Symptom: Client gets 500 with no useful message. Root cause: Bare except Exception without logging. Fix: Structured JSON logging + request_id for traceability.

Interview Prep

Three depths of explanation — 30s, 90s, 3min.
⚡ 30-Second Answer
I built a FastAPI service with 3 async endpoints (health check, readiness probe, chat) that proxies user prompts to Vertex AI Gemini. It uses Pydantic V2 for validation, implements exponential backoff retry on API calls, and is deployed on Cloud Run with a Docker container. The key pattern is asyncio.to_thread() to keep the event loop responsive while calling the synchronous Vertex AI SDK.
🎯 90-Second STAR Answer
S: For our applied AI module, I needed to build a production-ready API that serves Gemini responses without blocking under concurrent load.
T: The Vertex AI SDK is synchronous — calling it directly would freeze the FastAPI event loop, making the service unable to handle concurrent users. I also needed input validation, error resilience, and Cloud Run deployability.
A: I structured the app with 3 layers — routes, Pydantic schemas, and a Gemini service wrapper. I used asyncio.to_thread() to offload SDK calls to a thread pool. I added exponential backoff retry (1s → 2s → 4s + jitter) for transient API errors, and middleware for structured JSON logging with request IDs. The Dockerfile uses python:3.11-slim with layer caching.
R: The service now handles 50+ concurrent requests without blocking, with 99.9% uptime on Cloud Run. The retry mechanism recovers from quota errors automatically. The structured logging helped us debug a production issue within 2 minutes.
🔧 3-Minute Technical Walkthrough
Architecture: The request lifecycle is: Internet → Cloud Run HTTPS proxy → FastAPI ASGI server → Middleware (CORS, logging, error mapping) → Route handler → Pydantic validation → Service layer → Gemini API → Response JSON.

Key Decisions:
  • Async + to_thread(): The Vertex AI SDK is synchronous. Instead of making it async (which would require forking the SDK), we use asyncio.to_thread() which runs the blocking call in a thread pool executor. This keeps the event loop free for other requests.
  • Pydantic V2 field_validator: Migrated from V1's @validator to V2's @field_validator with @classmethod. It's faster, type-safe, and the modern pattern.
  • Retry with jitter: Only retries on ResourceExhausted (429) and ServiceUnavailable (503). Uses exponential backoff with random jitter to avoid thundering herd.
  • Health probes: /healthz for liveness (always 200), /readyz for readiness (checks Gemini connectivity). Cloud Run uses these for rolling updates and auto-healing.
Failure Modes:
  • Blocking the event loop — one slow call blocks all others. Fix: to_thread().
  • No retry — transient quota errors cause user-facing failures. Fix: backoff.
  • Missing GCP auth — Vertex AI returns 403. Fix: service account on Cloud Run.
