The Real Use Case (from group chat)

You have: 10-12 partitioned BigQuery tables. Some tables receive 20GB of data per day.

You need: One CSV file per table delivered to an external server (non-GCP). The pipeline runs nightly via Autosys scheduler.

The problem: Single CSV export from BQ is capped at ~1GB compressed. Even per-partition exports can exceed that for 20GB/day tables. Direct job.result() polling breaks if the scheduler node disconnects. Multiple wildcard files conflict with the "single file per table" requirement.

Key Design Decisions
  1. Single CSV per table — wildcard export creates multiple files, so we need a merge step
  2. 20GB/day per table — exceeds BQ's 1GB single-CSV limit, so partition-level iteration is needed
  3. Server destination (not GCP) — need network transfer to external server. Private network preferred if available
  4. Autosys scheduling — script must be self-contained, support checkpointing, clean exit codes

APPROACH A • RECOMMENDED FOR <50GB TABLES
📡 BigQuery Storage API — Stream Direct to Server
The Storage API reads table data as a stream of row batches. This bypasses the CSV export limit entirely — you read the raw data and write it to CSV on your server. No GCS intermediate needed. Works well for tables up to 50GB.
✅ Direct to server ✅ No GCS needed ✅ Single CSV output ⚠️ Memory: stream rows, don't load all

Single Table Export Script

# export_table.py — Stream BQ table directly to CSV on server # Uses BigQuery Storage API (read_session) for efficient streaming from google.cloud import bigquery_storage import csv, os, sys def export_table_to_csv(project, dataset, table, output_path, date_filter=None): """Stream BQ table to a single CSV file using Storage API. Never loads all data into memory — writes row-by-row. """ client = bigquery_storage.BigQueryReadClient() # Build the table path with optional partition decorator table_path = f"projects/{project}/datasets/{dataset}/tables/{table}" # Create a read session — specifies columns, format, etc. if date_filter: # Use SQL to filter by partition date filter_sql = f"SELECT * FROM `{project}.{dataset}.{table}` " \ f"WHERE _PARTITIONTIME = TIMESTAMP('{date_filter}')" read_session = client.create_read_session( parent=f"projects/{project}", read_session={ "data_format": bigquery_storage.DataFormat.AVRO, "table": table_path, "filter": {sql: filter_sql}, }, ) else: read_session = client.create_read_session( parent=f"projects/{project}", read_session={ "data_format": bigquery_storage.DataFormat.AVRO, "table": table_path, }, ) # Get the stream (one per worker — here using single) stream = read_session.streams[0] reader = client.read_rows(stream.name) # Read rows in batches and write to CSV rows_read = 0 with open(output_path, "w", newline="") as f: writer = None for batch in reader.rows(read_session): if writer is None: # First batch: set up CSV writer with column names fieldnames = [s.name for s in read_session.schema.schema] writer = csv.DictWriter(f, fieldnames=fieldnames) writer.writeheader() for row in batch: writer.writerow(row) rows_read += 1 return rows_read if __name__ == "__main__": # Example: export yesterday's partition rows = export_table_to_csv( project="my-project", dataset="my_dataset", table="orders", output_path="/data/exports/orders_20240115.csv", date_filter="2024-01-15" ) print(f"Exported {rows} rows to /data/exports/orders_20240115.csv")
Key: reader.rows() returns an iterator — data never all sits in memory at once. Each batch is ~10MB, written to the CSV file and discarded. For a 20GB table, peak memory usage is under 100MB.

Multi-Table Orchestrator

# run_all_exports.py — Orchestrate 10-12 tables with checkpoint import json, os from datetime import datetime, timedelta CONFIG = { "project": "my-project", "dataset": "my_dataset", "date": (datetime.now() - timedelta(days=1)).strftime("%Y-%m-%d"), "output_dir": "/data/exports", "checkpoint_file": "/data/exports/.checkpoint.json", "tables": [ {"name": "orders", "size_gb": 20}, {"name": "customers", "size_gb": 5}, {"name": "transactions", "size_gb": 12}, {"name": "products", "size_gb": 2}, {"name": "inventory", "size_gb": 8}, {"name": "payments", "size_gb": 15}, {"name": "returns", "size_gb": 1}, {"name": "shipments", "size_gb": 8}, ] } def load_checkpoint(): if os.path.exists(CONFIG["checkpoint_file"]): with open(CONFIG["checkpoint_file"]) as f: return json.load(f) return {"completed_tables": []} def save_checkpoint(completed): with open(CONFIG["checkpoint_file"], "w") as f: json.dump({"completed_tables": completed}, f) checkpoint = load_checkpoint() completed = set(checkpoint.get("completed_tables", [])) for table in CONFIG["tables"]: if table["name"] in completed: print(f"SKIP {table['name']} — already done") continue output_file = f"{CONFIG['output_dir']}/{table['name']}_{CONFIG['date']}.csv" print(f"EXPORT {table['name']} ({table['size_gb']}GB) → {output_file}") try: rows = export_table_to_csv( CONFIG["project"], CONFIG["dataset"], table["name"], output_file, date_filter=CONFIG["date"] ) print(f" Done: {rows} rows") completed.add(table["name"]) save_checkpoint(list(completed)) except Exception as e: print(f" FAILED: {e}") sys.exit(1) # Autosys detects non-zero exit → retry print("All tables exported successfully!")
The orchestrator runs each table sequentially. On failure, it exits non-zero → Autosys marks the job as failed. On retry, it skips completed tables. The "clear task" option resets the checkpoint if needed.
Storage API — Important Notes
  • Requires bigquery.readsession.googleapis.com API to be enabled
  • Billing: Storage API reads are charged per byte — ~$1.10/TB for US region
  • Network: Data goes through the internet (or VPC if server is on GCP)
  • Tables over 50GB: BQ may throttle — use GCS export approach instead
  • Complex column types (STRUCT, REPEATED) are serialized as JSON in Avro — flatten in SQL first
APPROACH B • BEST FOR TABLES >50GB
⚡ GCS Wildcard Export + Download + Server-Side Merge
Use BQ's native export to GCS with wildcard sharding (fast — parallel writes), then download all shards to your server and merge them into one CSV. This is the most robust approach for 20GB/day tables because BQ handles the heavy lifting.
✅ BQ handles distribution ✅ Works for any size table ⚠️ Needs GCS bucket ⚠️ Cleanup shards after merge

Step 1: Export to GCS with wildcard

# Step 1: Export one table to GCS as wildcard shards from google.cloud import bigquery client = bigquery.Client() # Export to wildcard — BQ splits into ~1GB shards table_ref = "my-project.my_dataset.orders$20240115" dest = "gs://my-bucket/exports/orders_20240115_*.csv" job = client.extract_table( table_ref, dest, job_config=bigquery.ExtractJobConfig( destination_format=bigquery.DestinationFormat.CSV, print_header=True ) ) job.result() print(f"Exported to wildcard. Files in GCS:") # List the generated files for obj in client.list_blobs("my-bucket", prefix="exports/orders_20240115_"): print(f" {obj.name} ({obj.size/1e6:.1f} MB)")

Step 2: Download + Merge on Server

# download_and_merge.sh — Run on the external server # Assumes gcloud auth is configured on the server TABLE="orders" DATE="20240115" BUCKET="gs://my-bucket/exports" LOCAL_DIR="/data/exports" # Step 2a: Download all shards gsutil -m cp "${BUCKET}/${TABLE}_${DATE}_*.csv" "${LOCAL_DIR}/" # Step 2b: Merge into single CSV (keep headers from first file only) FIRST_FILE=$(ls "${LOCAL_DIR}/${TABLE}_${DATE}_"*.csv | head -1) OUTPUT_FILE="${LOCAL_DIR}/${TABLE}_${DATE}.csv" cp "$FIRST_FILE" "$OUTPUT_FILE" for f in "${LOCAL_DIR}/${TABLE}_${DATE}_"*.csv; do if [ "$f" != "$FIRST_FILE" ]; then tail -n +2 "$f" >> "$OUTPUT_FILE" fi done # Step 2c: Cleanup shards rm "${LOCAL_DIR}/${TABLE}_${DATE}_"*.csv # Step 2d: (Optional) Cleanup GCS shards to save cost gsutil -m rm "${BUCKET}/${TABLE}_${DATE}_"*.csv echo "Merged → ${OUTPUT_FILE}" wc -l "$OUTPUT_FILE"
The merge uses tail -n +2 to skip CSV headers on all shards after the first. For very large merges, use awk 'NR==1 || FNR!=1' pattern for better performance.
Network Optimization for Server Download

If server IS on GCP (same region as bucket): Use -o "GSUtil:parallel_process_count=1" with gsutil -m — download goes through Google's internal network, ~10x faster than internet, zero egress cost.

If server is NOT on GCP: Use gsutil -o "GSUtil:parallel_process_count=4" -m cp for parallel downloads. Expect ~100-200 Mbps over internet for large files. Consider using a GCE VM in the same region as a staging point, then rsync to external server.

APPROACH C • BEST FOR CHECKPOINT/RESUME WORKFLOWS
🔄 Partition-by-Partition with Checkpoint File
If a table has 30 days of partitions and you need to export all 30 days into one combined file, iterate partition-by-partition. Each partition is a small, fast export. If the job fails on partition 19, the checkpoint tells it to start from 19 — not 1.
✅ Granular checkpoint ⚠️ Sequential — slower for full table ✅ Each partition <1GB guaranteed

Iterate Partitions with Resume

from google.cloud import bigquery from datetime import date, timedelta import json, os client = bigquery.Client() def export_partition_range(project, dataset, table, start_date, end_date, output_dir): checkpoint_file = f"{output_dir}/.checkpoint_{table}.json" # Load checkpoint completed_dates = set() if os.path.exists(checkpoint_file): with open(checkpoint_file) as f: completed_dates = set(json.load(f).get("completed", [])) current = start_date while current <= end_date: day_str = current.strftime("%Y%m%d") if day_str in completed_dates: current += timedelta(days=1) continue table_ref = f"{project}.{dataset}.{table}${day_str}" part_file = f"{output_dir}/{table}_{day_str}.csv" print(f"Exporting partition: {day_str}") job = client.extract_table(table_ref, part_file) job.result() print(f" Done: {job.output_rows} rows") completed_dates.add(day_str) with open(checkpoint_file, "w") as f: json.dump({"completed": sorted(completed_dates)}, f) current += timedelta(days=1) # All partitions done — merge into single file output_file = f"{output_dir}/{table}_{start_date.strftime('%Y%m%d')}" \ f"-{end_date.strftime('%Y%m%d')}.csv" merge_csv_parts(output_dir, table, output_file) print(f"Merged → {output_file}") def merge_csv_parts(output_dir, table, output_file): """Merge daily CSV files into one. Keeps headers from first file.""" parts = sorted([f for f in os.listdir(output_dir) if f.startswith(table + "_") and f.endswith(".csv")]) with open(output_file, "w") as out: for i, part in enumerate(parts): with open(os.path.join(output_dir, part)) as f: if i == 0: out.write(f.read()) else: next(f) # skip header out.write(f.read()) os.remove(os.path.join(output_dir, part))
Each day's partition is exported as a separate CSV. On success, all are merged and the daily parts are cleaned up. The checkpoint file makes it resilient — if the server reboots, only uncompleted partitions are re-exported.

Autosys Scheduling

How to wire it up with the "clear task" pattern
Autosys Job Definition

Your Autosys JIL (Job Information Language) definition looks like this:

/* autosys_jil.txt — One job per table is cleaner than one big job */ insert_job: bq_export_orders job_type: CMD command: /data/scripts/export_table.sh orders machine: etl_server_01 owner: etl_user permission: gx,wx date_conditions: 1 days_of_week: mo,tu,we,th,fr,sa,su start_times: "02:00" max_run_alarm: 1800 /* 30 min alarm */ term_run_time: 3600 /* kill after 1 hr */ insert_job: bq_export_customers job_type: CMD command: /data/scripts/export_table.sh customers ... /* Add each table as a separate job for parallel execution */ /* Or use a box for sequential execution */ insert_job: bq_export_box job_type: BOX ... /* The "clear task" = resetting checkpoint */ insert_job: bq_export_clear_checkpoint job_type: CMD command: /data/scripts/clear_checkpoint.sh ...
Autosys Patterns That Matter Here
  • Per-table jobs: Define one Autosys job per table. If 2 tables fail, only those 2 get retried. Don't put all 10-12 in one script.
  • Box with dependencies: Use a BOX to group all exports. Set conditions so the downstream ETL only triggers after ALL exports succeed.
  • Clear task = reset checkpoint: The "clear task" option they mentioned = deleting the checkpoint file so the export reruns from scratch. Expose as a separate job: bq_export_clear_checkpoint.
  • Exit codes: Your Python script must sys.exit(1) on any failure. Autosys detects this and retries based on your term_run_time and max_run_alarm settings.
  • Logging: Redirect stdout/stderr to a dated log file for debugging: command: /data/scripts/export_table.sh orders >> /data/logs/orders_\`date +\%Y\%m\%d\`.log 2>&1
The "Clear Task" Pattern

When a job fails midway (e.g., 6 of 10 tables done), the operator can:

  1. Option 1 — Resume: Just rerun the job. The checkpoint file skips the 6 completed tables and starts from table 7.
  2. Option 2 — Clear & Rerun (the "clear task"): Delete the checkpoint file first (rm /data/exports/.checkpoint.json), then rerun. Forces a full re-export.
  3. Option 3 — Partial clear: Edit the checkpoint file to remove specific tables. Only those tables get re-exported.

Expose these as separate Autosys jobs so operators don't need to SSH into the server.

Network Considerations

Sending 100GB+ across servers — make it fast
Scenario Recommended Approach Expected Speed (20GB table)
Server on GCP, same region GCS export + private download via gsutil ⚡ 2-3 minutes (0 egress cost)
Server on GCP, different region GCS export + download (cross-region egress ⚠️) ⚡ 4-6 minutes (egress ~$0.12/GB)
Server on-prem / other cloud BQ Storage API streaming (no GCS hop) ⏱️ 8-15 minutes (internet bandwidth bound)
Server on-prem with VPN to GCP GCS export + VPN download (or Storage API via VPN) ⚡ 3-8 minutes (depends on VPN throughput)
Bandwidth Cheats
  • Compress before transfer: Export as GZIP instead of CSV (BQ supports compression=GZIP on export). 20GB CSV → ~2GB GZIP. Decompress on server.
  • Parallel downloads: Use gsutil -m cp with parallel_process_count=4 for multiple shards simultaneously.
  • Staging VM: If server isn't on GCP, spin up a cheap GCE VM in the same region as your bucket. Download to VM (fast, free), then rsync to external server overnight.
  • Interrupt resume: Use curl -C - or wget -c for HTTP downloads. gsutil cp auto-resumes on interruption.

Situation: Needed to export 10-12 partitioned BQ tables nightly to an external server as single CSV files. Some tables had 20GB/day of data. The client was using a Python BQ client with job.result(), and workstation disconnects were killing exports after an hour of work.

Task: Design a reliable pipeline that produces one CSV per table, handles up to 200GB of total data per night, survives disconnects, and integrates with Autosys scheduling.

Action: Replaced single-CSV exports with the BigQuery Storage API to stream data directly to the server (no GCS intermediate). Implemented a checkpoint-based orchestrator that processes tables sequentially and tracks completion in a JSON file. Each table's (or partition's) checkpoint survives server reboots. 3-Minute Technical Walkthrough

The core challenge: BigQuery's export API has a 1GB compressed limit for single CSV destinations because the distributed workers must serialize through one writer. For a 20GB table, this creates a bottleneck. Additionally, the Python client's job.result() polls the BQ API — a disconnect drops the polling loop, even though the server-side job continues.

Architecture choice: I chose the Storage API over GCS wildcard export for this use case. The Storage API reads table data as a stream of Avro batches (~10MB each). Each batch is iterated reader.rows(read_session) and written to the output CSV file incrementally. Peak memory stays under 100MB regardless of table size.

Checkpoint orchestrator: The orchestrator script loads a JSON checkpoint file on startup. For each table, it checks if that table is in the completed list. If not, it runs the export, appends the table name to the checkpoint, and saves. If the script is interrupted (server reboot, Autosys kill, network drop), rerunning it picks up from the last completed table. For partitioned tables spanning multiple days (e.g., month-end full load), we track individual partition dates in the checkpoint for even finer granularity.

Autosys integration: Each table is a separate Autosys job (not one monolithic job). This means a failure in table 7 doesn't block tables 8-12 from running (if they're independent) or the box condition ensures ordering. The "clear task" their operator mentioned is implemented as a separate job that deletes the checkpoint file — no SSH required.

Network optimization: When the server is on GCP, gsutil -m cp with private Google network achieves 500+ Mbps. For external servers, the Storage API avoids GCS egress costs ($0.12/GB) entirely — data flows directly from BQ storage to the client over encrypted internet. For very large datasets, we add GZIP compression at the BQ export level (20GB → ~2GB), transfer the compressed file, then decompress on the server.

The Takeaway

TL;DR: For exporting partitioned tables to an external server as single CSVs, don't fight BQ's single-file limit — work around it.

Your three options (in order):
👉 Storage API — stream directly to server, no GCS needed, memory-safe
👉 GCS wildcard + download + merge — fastest for BQ, but needs GCS + egress costs
👉 Partition-by-partition — best checkpoint/resume, ideal for Autosys retries

The "clear task" pattern = delete checkpoint file. Simple. Powerful.
