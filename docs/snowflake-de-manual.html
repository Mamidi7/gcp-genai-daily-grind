1. Architecture — Why Snowflake Is Different

Storage-Compute Separation · Virtual Warehouses · Micro-Partitions · Caching
Snowflake architecture = మన ఊరి గోడౌన్ (godown) system లా ఉంటది.
Storage = the godown where all goods (data) live. Compute = loading docks. You can have 3 separate docks (ETL team, BI team, Data Science) all pulling from the same godown simultaneously. Nobody waits for anyone. Pay rent for godown space. Pay workers only when they're on shift. Postgres is like a kirana shop — stock and counter in the same tiny room. You can't add more counters without moving the whole shop.

Snowflake is a cloud-native data warehouse built on a fundamental insight: storage and compute don't need to live together.

In Postgres, your data lives on disk attached to the same machine that runs queries. Need more compute? Scale vertically (bigger machine, expensive, ceiling) or horizontally (read replicas, operational headache). In Snowflake, data lives in cheap cloud object storage (S3/Azure Blob/GCS). Queries run on independent virtual warehouses that spin up, execute, and auto-suspend. Ten warehouses can query the same table simultaneously — zero locking, zero contention.

⚠ COMMON PITFALL
Adding more nodes doesn't fix bad SQL. A cartesian join on XL is still slow — just 4x more expensive.
💡 INTERVIEW FRAME
"Snowflake separates storage and compute so you can run concurrent workloads on shared data without contention. Virtual warehouses are independent clusters that start, run, and suspend — you pay only for active seconds."
[ KEY INSIGHT — WHY THIS MATTERS FOR COST ]
ETL takes 4 hours at night? Spin up an L warehouse. Shut it down when done. BI queries during the day? Run on a separate M warehouse. Data scientist runs an ad-hoc query? Their own XS warehouse. Each team gets isolated compute. You pay only for the seconds each warehouse is active. No idle capacity. No "the ETL is blocking my dashboard" complaints.
Micro-Partitions: How pruning works without indexes

Snowflake stores data in immutable 50-500 MB micro-partitions — columnar chunks automatically created on ingestion. Every partition's metadata records min/max values for every column. When you query WHERE order_date > '2026-01-20', Snowflake reads the metadata (not data), finds partitions whose max date is before your threshold, and skips them entirely. Zero bytes scanned for irrelevant partitions.

This is automatic — no CREATE INDEX, no REINDEX, no DBA maintenance. But it only works well if data is loaded in roughly sorted order. Random date loads span the full range ⇒ pruning useless ⇒ every query scans everything. The fix: add a clustering key.

Three Cache Layers (and when each helps)

1. Metadata Cache: Micro-partition statistics. Service Layer. Always active, zero cost.

2. Result Cache: Same query (byte-for-byte) within 24h returns cached result. Zero compute cost. WHERE date = '2026-05-27' and WHERE date = '2026-05-28' are different cache entries.

3. Warehouse Cache (SSD): Recently accessed micro-partitions stay on local SSD. Lost when warehouse suspends (default: 10 min idle).

⚠ YEPPUDU CONFUSE AITAV — MICRO-PARTITIONS vs CLUSTERING KEYS
Same thing kaadu. Micro-partitions are automatic immutable storage chunks (typically 50-500 MB) created for every table load. Clustering is an optional physical re-ordering strategy you choose for very large tables with selective filters. Rule: every table has micro-partitions; only some tables should have clustering keys.
🧠 MEMORY AID
3 Cache Layers = MRC: Metadata (free, always on), Result (24h exact-query reuse), Compute/SSD (fast but lost on suspend). Interview lo one-liner: “M is global brain, R is replay, C is local scratchpad.”
[ 30s INTERVIEW FRAME + REAL NUMBERS ]
“In my last Snowflake workloads, a 12 TB fact table with date filters scanned ~9.8 TB before clustering and ~1.4 TB after clustering on order_date, region (~85% reduction). I first rely on automatic micro-partition pruning, then add clustering only for high-selectivity tables because reclustering itself consumes credits.”

⚠ PITFALL — COPY INTO + PURGE = DATA LOSS
COPY INTO is idempotent by default. Snowflake tracks loaded files — running the same command twice won't duplicate data. But if you use PURGE = TRUE, files are deleted from S3 after the load commits. If the load fails mid-way? Already-purged files are gone. Recommendation: use PURGE only after verifying row counts. Better: separate archiving step.
⚠ YEPPUDU CONFUSE AITAV — SNOWPIPE vs SNOWPIPE STREAMING
Snowpipe = file-based ingestion (S3 event → load). Snowpipe Streaming = row-based ingestion via SDK for low-latency events. Wrong choice hurts: file trickle into Snowpipe Streaming can overpay; row-by-row events in classic Snowpipe can under-deliver latency SLAs.

🧠 MEMORY AID
Ingestion mnemonic: BERS — Batch (COPY), Event-file (Snowpipe), Row-stream (Snowpipe Streaming), State-change (Streams+Tasks).
[ 30s INTERVIEW FRAME ]
“I choose ingestion by latency and payload shape: COPY INTO for predictable batch (100-500 GB/night), Snowpipe for event-driven files (~1 min), Snowpipe Streaming for row-level low latency, Streams+Tasks for CDC transforms, and Dynamic Tables when team wants declarative refresh over handwritten MERGE logic.”
[ LISTING 001: COPY INTO — BATCH LOAD FROM S3 ]-- File format (tells Snowflake how to parse) CREATE FILE FORMAT my_csv TYPE='CSV' FIELD_OPTIONALLY_ENCLOSED_BY='"' SKIP_HEADER=1 NULL_IF=('NULL','null',''); -- External stage (pointer to cloud storage) CREATE STAGE my_s3 URL='s3://my-bucket/landing/' STORAGE_INTEGRATION=my_s3 FILE_FORMAT=my_csv; -- Load: auto-decompresses gzip, validates, idempotent COPY INTO raw.orders FROM @my_s3 PATTERN='.*orders_.*\\.csv\\.gz' ON_ERROR='SKIP_FILE_3%' PURGE=TRUE; -- ⚠ deletes from S3 after load!

Streams + Tasks: CDC Within Snowflake

A Stream is a lightweight CDC log over a table. Every DML (INSERT/UPDATE/DELETE) is recorded with metadata columns: METADATA$ACTION, $ROW_ID, $IS_UPDATE. A Task is a scheduled SQL executor that consumes the stream.

[ LISTING 002: STREAM + TASK — INCREMENTAL CDC ]-- Create a stream over the source table CREATE STREAM orders_stream ON TABLE raw.orders; -- Create a task that runs every 5 minutes CREATE TASK process_orders WAREHOUSE = etl_wh SCHEDULE = '5 minute' WHEN SYSTEM$STREAM_HAS_DATA('orders_stream') AS MERGE INTO dw.fact_orders t USING ( SELECT order_id, customer_id, amount, METADATA$ACTION AS action FROM orders_stream ) s ON t.order_id = s.order_id WHEN MATCHED AND s.action = 'DELETE' THEN DELETE WHEN MATCHED THEN UPDATE SET t.amount = s.amount WHEN NOT MATCHED THEN INSERT (order_id, customer_id, amount) VALUES (s.order_id, s.customer_id, s.amount);
🔑 IDEMPOTENCY
COPY INTO tracks which files it loaded via metadata. Running it twice on same stage = zero duplicates. Safety net for scheduled ETL.
⚠ STREAM PITFALL
Streams are offset-based. Once consumed by a DML statement, the offset advances. Two tasks reading the same stream = second task sees empty stream. Use separate streams for separate consumers.

3. SQL — The Patterns You Write Every Week

QUALIFY · MERGE · CTEs · LAG/LEAD · Window Functions

These four patterns cover ~80% of production SQL. Master them and you're ahead of most candidates claiming "5 years Snowflake experience." Each pattern below answers: what problem it solves, where it sits in the pipeline, and when NOT to use it.

Window functions ante train lo seat numbers allocate chesinattu.
Entire train = full dataset. Each coach = PARTITION BY. Inside each coach, seat order = ORDER BY. ROW_NUMBER() gives each passenger a rank without removing anyone. After ranks are assigned, QUALIFY says “each coach lo first seat only ivvu.”
⚠ YEPPUDU CONFUSE AITAV — WHERE vs HAVING vs QUALIFY
Order gurthu pettuko: WHERE filters rows before grouping, HAVING filters after grouping, QUALIFY filters after window functions. Writing trick: write WHERE first, then ask “aggregate condition aa?” → HAVING, “window rank condition aa?” → QUALIFY.
🧠 MEMORY AID
WHQ Ladder: WHERE (raw rows) → HAVING (group result) → QUALIFY (window result). Interview lo quick line: “Q comes last because window comes late in logical execution.”
[ 30s INTERVIEW FRAME + REAL NUMBERS ]
“For weekly dedup + CDC pipelines, I use QUALIFY ROW_NUMBER for latest-record selection, LAG for change detection, MERGE for atomic upsert, and CTEs for readability. Example: in a 180M-row order table, QUALIFY-based dedup reduced output to 132M unique orders before downstream joins, cutting runtime from 19 min to 7 min on a Medium warehouse.”
[ WHEN NOT TO USE — QUICK GRID ]
Dedup with QUALIFY: don’t use when full history is needed. LAG-based CDC: don’t use if source already ships reliable CDC flags. MERGE: don’t use for pure append-only loads. Long CTE chains: don’t use when same intermediate output is reused by multiple downstream jobs — materialize once.
[ PATTERN 001: DEDUPLICATION — LATEST ROW PER ENTITY ]-- Problem: raw_orders has duplicates (same order_id loaded multiple times) -- Solution: keep only the latest version per order_id SELECT * FROM raw_orders QUALIFY ROW_NUMBER() OVER ( PARTITION BY order_id ORDER BY loaded_ts DESC ) = 1; -- QUALIFY = WHERE for window functions -- Snowflake + BigQuery support it. For other DBs, wrap in CTE + WHERE rn=1. -- When NOT to use: if you need ALL versions (SCD Type 2), use LAG() instead.
[ PATTERN 002: CHANGE DETECTION — CDC WITH LAG() ]-- Problem: which customers changed their address? -- Solution: compare current row to previous row per customer SELECT customer_id, address, effective_date, LAG(address) OVER ( PARTITION BY customer_id ORDER BY effective_date ) AS prev_address, CASE WHEN address IS DISTINCT FROM LAG(address) OVER (PARTITION BY customer_id ORDER BY effective_date) THEN 'CHANGED' ELSE 'SAME' END AS status FROM customer_history; -- Only rows with status='CHANGED' → feed into SCD Type 2 MERGE -- When NOT to use: if source already provides CDC flags (use those instead)
[ PATTERN 003: MERGE — UPSERT (MATCHED→UPDATE, NOT MATCHED→INSERT) ]MERGE INTO dim_customer AS t USING stg_customer_updates AS s ON t.customer_id = s.customer_id WHEN MATCHED THEN UPDATE SET t.name=s.name, t.address=s.address, t.updated_ts=CURRENT_TIMESTAMP() WHEN NOT MATCHED THEN INSERT (customer_id, name, address, created_ts, updated_ts) VALUES (s.customer_id, s.name, s.address, CURRENT_TIMESTAMP(), CURRENT_TIMESTAMP()); -- MERGE is atomic (all-or-nothing). Max 3 WHEN clauses. -- When NOT to use: if you only INSERT (use INSERT), only UPDATE (use UPDATE)
[ PATTERN 004: CTE PIPELINE — STAGE-BY-STAGE TRANSFORMATION ]WITH source AS ( SELECT * FROM raw.orders WHERE status IS NOT NULL AND amount > 0 ), deduped AS ( SELECT * FROM source QUALIFY ROW_NUMBER() OVER ( PARTITION BY order_id ORDER BY loaded_ts DESC ) = 1 ), enriched AS ( SELECT d.date_key, c.customer_key, p.product_key, o.amount, o.qty FROM deduped o LEFT JOIN dim_date d ON o.order_date = d.date LEFT JOIN dim_customer c ON o.customer_id = c.customer_key LEFT JOIN dim_product p ON o.product_id = p.product_key ) SELECT date_key, customer_key, COUNT(*) AS orders, SUM(amount) AS revenue FROM enriched GROUP BY date_key, customer_key; -- CTEs make pipelines readable. Each block = one stage. -- When NOT to use: if intermediate results used by 3+ downstream queries -- → materialize as a temp table instead (CTE runs once per reference)

4. Data Modeling — Design for Queryability

Star Schema · SCD Types · Clustering Keys · Naming Conventions
Star schema = మన లైబ్రరీలో పుస్తకాల అమరిక లా ఉంటది.
Fact table = the checkout register recording every book borrowed (many rows, tall). Dimension tables = the shelves sorted by genre (history shelf, science shelf, Telugu shelf — short, wide). Queries are fast because you know exactly which shelf to look at. No need to scan the entire library.

Star Schema Basics

Fact tables contain measures (amount, qty, count) and foreign keys to dimensions. They grow tall — billions of rows. Dimension tables contain descriptive attributes (customer name, address, product category). They grow wide — dozens of columns, fewer rows.

[ SNOWFLAKE-SPECIFIC: CLUSTERING KEYS ]
Snowflake automatically partitions data into micro-partitions. But if your fact table has random date distribution, every query will scan everything. A clustering key enforces physical sort order, improving partition pruning. Example: ALTER TABLE fact_orders CLUSTER BY (order_date). Re-clustering costs credits — only cluster large tables (>1 TB) with selective queries.

When to use: Tables >1 TB, queries have highly selective WHERE clauses on a date/region column.
When NOT to use: Small tables (<100 GB), tables queried without filters, tables that are constantly bulk-loaded.

SCD Type 2 in Snowflake

Slowly Changing Dimension Type 2 tracks history by creating a new row for every change. Snowflake makes this easier with Streams + Tasks and QUALIFY + LAG for change detection.

SCD types ni ration card address update laga think cheyyi.
Type 1 = old address erase చేసి కొత్తది రాయడం (no history). Type 2 = every move ki new page open చేసి old page archive చెయ్యడం (full history). Type 3 = current + previous address మాత్రమే ఉంచడం (limited history columns).
⚠ YEPPUDU CONFUSE AITAV — CLUSTERING vs PARTITIONING
Traditional partitioning physically splits data into explicit partitions/files you define. Snowflake mainly uses automatic micro-partitions; clustering just improves data locality within those micro-partitions. So design decisions are different: you tune clustering depth, not manual partition DDL.
🧠 SCD MEMORY AID
SCD = 1/2/3 → 1=Replace, 2=History, 3=Selected columns. Quick interview line: “If compliance asks ‘as-of report’, Type 2 is default.”
[ WHEN NOT TO USE ]
Type 1: don’t use when audits require historical truth. Type 2: don’t use for ultra-high-churn attributes unless storage/query plan supports growth (e.g., 50M customers × 20 changes/year). Type 3: don’t use if business asks for more than “current + previous” versions.
[ 30s INTERVIEW FRAME + REAL NUMBERS ]
“I model reporting marts in star schema: facts at transaction grain, dimensions for attributes, and SCD type based on history needs. In one pipeline, customer dimension grew from 12M to 38M rows in 18 months due to Type 2 history, but enabled accurate month-end re-statement reports and reduced reconciliation defects by ~60%.”
[ PATTERN 005: SCD TYPE 2 — TRACK HISTORY ]MERGE INTO dim_customer_scd2 AS t USING ( SELECT customer_id, name, address, LAG(address) OVER ( PARTITION BY customer_id ORDER BY loaded_ts ) AS prev_address FROM stg_customer ) s ON t.customer_id = s.customer_id AND t.is_current = TRUE WHEN MATCHED AND s.address IS DISTINCT FROM t.address THEN UPDATE SET t.is_current = FALSE, t.end_date = CURRENT_DATE(); -- Then insert the new version (separate INSERT for clarity) INSERT INTO dim_customer_scd2 (customer_id, name, address, is_current, start_date, end_date) SELECT customer_id, name, address, TRUE, CURRENT_DATE, NULL FROM stg_customer s WHERE NOT EXISTS ( SELECT 1 FROM dim_customer_scd2 t WHERE t.customer_id = s.customer_id AND t.address = s.address AND t.is_current = TRUE );
⚠ SCD COST
SCD Type 2 grows dimension tables without bound. A customer with 200 changes creates 200 rows. Plan for this: partition history by year, archive old rows.
💡 INTERVIEW FRAME
"I use SCD Type 2 for slowly changing attributes when business needs historical reporting. Type 1 for reference data that doesn't need history."

5. Performance — Faster Queries, Lower Cost

Warehouse Sizing · Query Profiling · Materialized Views · Search Optimization
Query optimization = కూరగాయలు కోసేటప్పుడు కత్తి పదును చెక్ చేసుకోవడం లా ఉంటది.
A blunt knife cuts vegetables slowly — you apply more force, get tired, waste energy. A sharp knife cuts clean. Similarly, bad SQL + bigger warehouse = still slow, just more expensive. Sharpen the query first, THEN scale. Snowflake's query profile is your whetstone.

Warehouse Sizing Tradeoffs

SizeNodesCredits/hrBest For
X-Small11Dev, light ad-hoc queries
Small22BI dashboards, small ETL
Medium44Production ETL, moderate joins
Large88Large transformations, heavy queries
X-Large1616Very large scans, complex aggregations
⚠ CRITICAL: Warehouses scale horizontally, not vertically
Doubling warehouse size (XS → S) doubles nodes. It does not make a single-threaded query faster. Queries must use parallel operators (scan across partitions, distributed joins) to benefit. Always check query profile first.

Query Profiling: The First Thing To Do

After running a slow query, click Query Profile in Snowsight. Look for:

  1. Full table scans on large tables — indicates poor partition pruning. Check clustering.
  2. Join spill — data exceeds memory, spills to remote disk. Increase warehouse size or add cluster key.
  3. Heavy aggregation — GROUP BY on high-cardinality column. Pre-aggregate with materialized views.
  4. Excessive data spilling — warehouse is too small for the data volume.
✓ Profile first, then size up
✗ Sizing up without profiling wastes credits
✓ Clustering helps selective queries
✗ Clustering costs credits to maintain

Materialized Views

Snowflake's materialized views store pre-computed results and auto-refresh when base data changes. Use them for:

  • Pre-aggregated summaries (daily totals, counts by category)
  • Filtered subsets of large tables
  • Expensive joins across stable data
[ LISTING 006: MATERIALIZED VIEW — PRE-AGGREGATE DAILY SALES ]CREATE MATERIALIZED VIEW daily_sales_mv AS SELECT order_date, SUM(amount) AS total_revenue, COUNT(*) AS order_count FROM fact_orders GROUP BY order_date; -- MV is auto-refreshed when fact_orders changes -- Limitations: only a single table, no WHERE, no JOIN, no HAVING -- Cost: credits for refresh. Only worth it for large tables with expensive aggregations.
⚠ YEPPUDU CONFUSE AITAV — WAREHOUSE SIZE MYTH
Doubling warehouse size doesn't guarantee a single-threaded query gets faster. Idi supermarket lo extra checkout counters add chesinattu — queue unte chala help, oka customer maatrame unte use ledu. Scale helps parallel workloads, not fundamentally poor plan shapes.
🧠 PROFILE CHECKLIST
FJHA mnemonic: Filter pushdown unda? Join spill jaruguthunda? Heavy aggregation ekkada? Amount scanned entha? Start here before resizing warehouses.
[ WHEN NOT TO USE ]
Bigger warehouse: don’t use as first fix for unfiltered scans. Materialized View: don’t use for rapidly changing base tables where refresh cost exceeds query savings. Search Optimization: don’t enable on low-selectivity predicates (you pay maintenance without gains).
[ 30s INTERVIEW FRAME + REAL NUMBERS ]
“My performance workflow is profile-first: FJHA checklist, then SQL rewrite, then storage tuning, and only then warehouse resize. In one workload scanning ~4.6 TB/query, adding selective filters + clustering cut scan to ~620 GB and runtime from 11m to 2m40s; we kept warehouse at Medium instead of scaling to XL, saving ~70% credits.”
🔍 QUERY PROFILE
Run EXPLAIN USING JSON before the query to see the plan without executing.
⚠ MV LIMITS
Materialized views in Snowflake are limited: single table, no JOIN, no WHERE, no HAVING, no window functions. Use them only for simple aggregations.
